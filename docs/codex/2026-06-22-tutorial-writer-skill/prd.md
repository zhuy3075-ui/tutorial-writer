# 教程课本 Skill PRD

## 背景与目标

`tutorial-writer` 是一个面向线下 AI 编程培训场景的教程生成 Skill。它要把知识库素材整理成结构化、通俗、可讲、可练、可发布到飞书文档的教程草稿。

当前作者的手工流程是：从知识库抽取内容，组织语言，设计截图，放进飞书排版。最耗时、最不稳定的部分是语言通俗化和配图策略，容易靠感觉写，导致教程风格不统一、结构不完整、课堂练习不够落地。

本次目标不是创建通用 Skill 方法论，而是完善现有 `tutorial-writer` Skill 的制作文档，让后续实现者能按清晰边界整理 Skill 文件、验证质量，并用真实教程试跑。

## 用户画像

### 线下学员

- 有 AI 编程基础，用过 prompt、IDE，跑通过项目。
- 不是纯小白，但常卡在“能跑但不懂原理”“能抄但不会变通”。
- 需要教程快速进入实操，不需要长篇行业背景。
- 需要课堂内能完成、能验证的练习，而不是模糊的“回去试试”。

### 教程作者

- 是知识博主或内容创作者，手里已有知识库素材。
- 想做体系化教程，但不想每篇都重新摸索语言、截图、飞书排版。
- 反感晒成绩不讲资源差、干货藏半截、俯视小白、内容不落地、前奏太长、半收费钓鱼式教程。
- 内容立场是“别人给结论，你给路径”。

## 核心痛点

1. 语言通俗化不稳定  
   专业概念容易直接堆术语，缺少“先例子后定义”和生活化类比。

2. 配图策略靠经验  
   不清楚什么时候必须截图、怎么标注，容易该截不截或用动图替代静态说明。

3. 飞书排版不成体系  
   容易整篇纯文字，或者只用一种格式，浪费 Callout、表格、代码块、折叠块的结构能力。

4. 教程缺课堂可验证闭环  
   练习和作业容易没有材料、没有合格标准、没有讨论节点。

5. Skill 结构需要持续保持一致  
   当前参考文件已放在 `tutorial-writer/references/` 下，且 `SKILL.md` 已按 `references/*.md` 引用。后续修改时必须保持目录和引用路径一致。

## 产品范围

### In Scope

- 明确 `tutorial-writer` 的目标读者、作者场景和教程质量标准。
- 定义从知识库素材到飞书教程草稿的生成流程。
- 固化三种教程骨架：操作指南型、概念讲解型、案例拆解型。
- 固化四段式知识点展开：为什么需要、怎么做、最容易犯的错、一个练习。
- 明确截图策略、飞书排版规则、课堂练习和讨论节点要求。
- 给出后续整理现有 Skill 文件的执行计划和验收标准。

### Out of Scope

- 不生成通用“如何制作所有 Skill”的方法论。
- 不扩展到通用 Skill 方法论或无关 Skill。
- 不新增脚本、自动化校验器或飞书 API 集成。
- 不负责真实截图制作，只在教程草稿中标注配图位。
- 不负责发布到飞书，只生成适合复制进飞书的 Markdown 草稿。

## 成功标准

### 白话版

- 作者拿到知识库素材、教程主题和目标受众后，可以按 Skill 输出一篇能在课堂讲、能让学员动手做的教程草稿。
- 学员在前五行能知道时间、目标、准备和环境。
- 教程前 200 字内进入动手环节，不用先读长篇背景。
- 每个关键知识点都有解释、步骤、踩坑和练习。
- 教程里能看出什么时候该截图、截图要标什么。
- 结尾有 5 分钟内可验证的练习，以及前置阅读和下一步。

### 技术版

- `SKILL.md` 保持主流程精简，详细规则放入 reference 文档。
- `writing-rules.md`、`tutorial-structure.md`、`screenshot-guide.md`、`feishu-format.md` 四类规则均被 Skill 工作流引用。
- 输出教程包含配图位、练习、讨论节点、来源引用和飞书排版四件套。
- 后续整理后的 Skill 能通过基础结构校验，并至少用 3 篇真实教程主题试跑。

## 风险与约束

- 规则太多会导致 Skill 输出僵硬；主流程必须保持短，细节按需加载。
- 目前没有真实教程样例作为回归基准；第一轮验收必须靠 3 篇真实主题试跑补足。
- 若后续移动 reference 文件或修改 `SKILL.md` 引用路径，必须同步更新另一侧，避免执行者按旧路径查找失败。
- 不引入脚本是刻意取舍：当前问题主要是写作流程和内容标准，先用文档约束更简单。
