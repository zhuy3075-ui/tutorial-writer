# tutorial-writer

从知识库素材到飞书教程的全流程生成器。目标读者：有 AI 编程基础的线下培训学员——能跑项目但卡在"能抄不会变通"。

## 输出边界

- 默认交付物是**已发布的飞书云文档（docx）**，不是本地草稿，也不是 Drive 里的普通 `.md` 文件。
- 除非用户明确说“只生成草稿 / 不上传 / 不调用飞书 CLI”，否则必须实际执行 `lark-cli docs +create --api-version v2` 创建飞书云文档，并返回 `/docx/` 链接。
- 发布教程正文时必须使用 DocxXML 表达高亮块、表格、代码块、检查项、分栏、画板等富文本结构；不要用 Markdown 上传替代漂亮排版。
- 只有当用户明确要求“上传 Markdown 文件 / 原生 .md 文件 / Drive 普通文件”时，才使用 `lark-cli markdown +create`。
- 每篇教程至少规划 1 个画板类图，并按内容适配流程图、思维导图、对比图、层级图、泳道图、决策树或闭环图。画板图必须层次分明、美观，不允许只用纯文字代替关键流程图。
- 教程正文必须有知识点之间的逻辑衔接、必要解释和过渡，不允许把知识库摘录拼接成章节。
- 教程要参考高质量公众号长文的图文节奏：先用自然语言讲场景、问题和判断，再给必要操作；少用代码堆砌，不把教程写成命令清单。
- 对快速变化、知识库信息不足、用户要求补充资料的主题，必须适度联网搜索并引用可靠来源。
- 联网搜索用于校准教程内容，不用于自动优化、重构或改写知识库仓库。
- 正式写正文前必须先判断教程形态：单节教程、小合集还是整套课程；再决定放在单篇飞书文档、多篇文档，还是 Wiki 知识库结构。
- 可在合适位置插入由 img2.0 生成的上下文相关图片，用于缓解学习压力、知识点演示和阶段汇总；一旦规划为“需要生成”，必须实际生成图片、保存本地资产、用 `lark-cli docs +media-insert` 插入飞书文档，并单独创建生图提示词飞书文档。

## 工作流

收到教程主题后，按以下流程执行：

```
判断教程范围：单节 / 小合集 / 整套课程
    ↓
规划飞书承载方式：单文档 / 多文档 / Wiki 知识库
    ↓
知识库提取核心知识点（3-5 个）
    ↓
标记陈旧风险，必要时联网校准定义 / 版本 / 案例 / 限制
    ↓
多 Agent 深度分析：内容主线 / 视觉图表 / 教学落地
    ↓
搭建知识逻辑链：读者当前状态 → 中间台阶 → 最终能力
    ↓
设计图文节奏：自然语言解释 → 关键图/表 → 必要操作 → 读图/判断
    ↓
选定教程骨架（tutorial-structure.md 四种之一）
    ↓
把每个知识点的四段式内容嵌入所选教程骨架，并写清过渡
    ↓
按 writing-rules.md 逐条改写语言
    ↓
规划截图位与画板图（按 screenshot-guide.md 和 feishu-format.md 规则）
    ↓
规划 img2.0 插图位与生图提示词文档（按 image-illustration.md 规则）
    ↓
生成 DocxXML 富文本正文（按 feishu-format.md 规范）
    ↓
执行飞书 CLI 创建云文档并验证 /docx/ 链接
    ↓
如有 img2.0 图片：生成图片 → 插入飞书文档 → 创建提示词文档
```

## 四段式展开

每个知识点都要覆盖以下四类信息，但不要另起一套重复结构；把它们嵌入 `references/tutorial-structure.md` 选定的教程骨架中：

1. **为什么需要这个**——用读者已有经验切入，1-2 句话
2. **怎么做**——分步骤，每步可独立验证
3. **最容易犯的错**——必须有至少一个你亲手踩过的坑
4. **一个练习**——5 分钟内可在课上完成并验证

## 需要加载的 reference

- 判断教程范围、章节部署和飞书承载方式时，读 `references/course-planning.md`
- 选教程骨架时，读 `references/tutorial-structure.md`
- 规划知识逻辑链、联网补充、过渡和解释深度时，读 `references/content-depth.md`
- 改写语言时，读 `references/writing-rules.md`
- 标注配图位时，读 `references/screenshot-guide.md`
- 规划 img2.0 生成图、提示词文档和飞书插入位置时，读 `references/image-illustration.md`
- 输出飞书云文档或飞书草稿前，读 `references/feishu-format.md`

## 多 Agent 分析要求

正式写教程正文前，先调用多 Agent 做深度分析。至少拆成 2 个独立视角：

1. **内容主线 Agent**：检查知识点是否有先后关系、因果关系、过渡、解释深度和外部资料缺口。
2. **视觉教学 Agent**：检查哪些内容适合流程图、思维导图、对比图、层级图、泳道图或决策树，并给出画板结构建议。

如果主题涉及具体工具/API/平台，再增加 **资料校准 Agent**：联网搜索官方资料、版本差异、限制和可靠案例。

主 Agent 必须综合这些分析后再写教程；不要把多个 Agent 的结论原样堆进正文。

## 读者基线

- 有 AI 编程使用经验（用过 prompt、IDE、跑通过项目）
- 线下课堂环境，教程是讲师的脚本
- 不需要解释 AI 基础概念，但领域专属概念仍需零术语起步
- 练习必须课堂内可完成，练习后埋讨论节点

## 输出检查清单

每篇教程输出前，确认：
- [ ] 已判断教程形态：单节 / 小合集 / 整套课程，并说明理由
- [ ] 已规划承载方式：单文档 / 多文档 / Wiki 知识库，并说明理由
- [ ] 已规划章节与小节部署；如果是合集或课程，先有总框架再写小节
- [ ] 选定了一种骨架且结构完整（开头/正文/结尾）
- [ ] 如果是项目复盘或工具体验分享，优先考虑“公众号项目实战型”骨架：先展示成果，再拆流程和方法
- [ ] 正式写正文前已完成多 Agent 深度分析，并吸收进大纲
- [ ] 已标记知识库中可能陈旧的知识点；必要时已联网校准并标注来源用途；若未联网，原因明确
- [ ] 联网校准只用于教程内容，没有改写知识库仓库
- [ ] 已建立知识逻辑链：读者当前状态 → 中间台阶 → 最终能力
- [ ] 已设计图文节奏：自然语言解释为主，图片/画板/表格服务理解，代码只在必要处出现
- [ ] 每个知识点四段式完整
- [ ] 每个知识点之间都有过渡，不是材料拼接
- [ ] 代码块、表格、画板、图片、高亮块之间都有承接说明；不允许代码块后直接硬切到图表或评分表
- [ ] 核心知识点至少覆盖场景、问题、原理、操作、判断
- [ ] 通过了 writing-rules.md 全部适用规则
- [ ] 标题层级和序号层级正确：H1/H2/H3 不跳级；有序列表不混乱；子步骤缩进清晰
- [ ] 配图位已标注（不生成图，只标注位置和描述）
- [ ] 至少 1 个画板类图已规划，且说明图类型、层级、节点和视觉要求；图表类型与内容匹配
- [ ] 已判断是否需要 img2.0 插图；需要时已规划插入位置、教学目的、图片提示词和本地资产路径
- [ ] 已生成独立的 img2.0 生图提示词文档；如实际发布，提示词文档也作为独立文档或附件交付
- [ ] 对比内容使用表格、分栏、对比图或高亮块显著显示
- [ ] 飞书排版四件套已使用；如实际创建飞书文档，使用 DocxXML 富文本块而不是普通 Markdown 文件上传
- [ ] 已实际执行 `lark-cli docs +create --api-version v2` 创建飞书云文档；除非用户明确要求只生成草稿
- [ ] 飞书正文不是普通 Markdown 导入，必须能看到高亮块、表格、代码块、检查项、分栏/画板等富文本块
- [ ] 场外文档出处置顶
- [ ] 练习可在 5 分钟内验证
- [ ] 至少一个讨论节点
- [ ] 实操作业（如章节需要）：任务 + 现成材料（prompt/命令/模板），可直接复制使用
- [ ] 返回的是 `/docx/` 云文档链接，并用 `docs +fetch --api-version v2` 验证标题和富文本块存在
- [ ] 如规划了 img2.0 图片，图片已生成并插入飞书文档；生图提示词文档也已创建成飞书文档或明确交付
