# writing-rules.md — 教程写作铁律

每篇教程写完后逐条自检。不是"建议"，是铁律。

---

## 一、通俗语言（6 条）

### 1. 零术语起步

专业词第一次出现必须配生活化类比。类比没写，不准往下走。

**好**：你想把模糊照片变清晰对吧？这个技术叫"图像超分辨率"——就像把马赛克还原成高清照片。

**烂**：图像超分辨率是指通过算法将低分辨率图像重建为高分辨率图像。

### 2. 先例子后定义

先让读者看见具体场景，再给抽象概念。顺序反了读者会关掉。

**好**：你写了一段 prompt，GPT 给了个答案，但答案不对。你再改 prompt，它又给另一个答案。你反复试了十次——这个过程叫"迭代优化"。

**烂**：迭代优化是 AI 编程中的核心方法论，指通过多次反馈循环逐步逼近目标结果。

### 3. 一段不超过四行

超过四行的段必须拆。连续三段纯文字且无视觉断点（加粗/列表/图片），强制插入断点。

### 4. 用"你"说话

全篇用"你"，不用"用户""读者""学习者"。

**好**：你打开终端，输入这个命令，你会看到——

**烂**：用户应打开终端并执行以下命令。

### 5. 每 300 字塞一个"试试看"

读完一段就能动手，不是读完三篇才能动手。

```
试试看：复制上面那段代码，改一下 system prompt，看看输出有什么变化？
```

### 6. 难点预告，不突击

遇到较难的概念，先给心理锚点再展开。

**好**：这一段会有点绕，但熬过去你就理解 Agent 是怎么"思考"的了。深呼吸，我们开始。

**烂**：直接甩技术细节，没有任何缓冲。

### 6.1 知识点必须有过渡

每个大段落开头必须交代“为什么现在讲这个”。教程不是知识卡片集合，不能把知识库材料按顺序拼起来。

**好**：上一节你已经能创建文档了，但如果不知道怎么检查结果，就会误以为“命令没报错=成功”。所以这一节先讲验证。

**烂**：下一节：验证方法。

### 6.2 解释不能只停在“是什么”

核心知识点至少覆盖：场景、问题、原理、操作、判断。

少于这五层时，读者会出现“看懂了，但不会用”的感觉。

**好**：先说什么时候会遇到它，再说不懂会卡哪里，再用类比解释，再给步骤，最后给判断标准。

**烂**：先定义，再贴命令，然后进入下一节。

### 6.3 代码块和图表之间必须有桥

代码块、表格、画板、图片、Callout 之间不能硬切。每个块出现前后都要回答一个问题：为什么现在看它？

特别是代码块后面，不能直接接评分表、对比表、流程图或画板。中间必须加 1-2 句桥接：

1. 这段代码刚刚证明了什么。
2. 下一张表/图要帮你判断什么。

**好**：

```text
先看这段最小配置。它不是完整项目，只负责展示“什么时候触发上传、什么时候应该失败”。
```

```bash
lark-cli docs +create --api-version v2 --as user --content @./tutorial.xml --json
```

```text
命令能跑通，只说明文档创建成功；它还不能证明教程好不好用。接下来用这张检查表，把“能创建”和“适合教学”拆开判断。
```

| 检查项 | 合格标准 | 不合格时先改什么 |
|--------|----------|------------------|
| 链接形态 | 包含 /docx/ | 先改上传方式 |
| 富文本块 | 能看到 callout/table/pre | 先改 DocxXML |

**烂**：

```bash
lark-cli docs +create ...
```

| 评分项 | 分数 |
|--------|------|
| 排版 | 80 |

上面这种写法的问题是：读者不知道表格为什么突然出现，也不知道它和代码有什么关系。

块间桥接模板：

```text
上面的代码解决的是 ______；但要判断它是否适合课堂，还要看 ______。
```

```text
这张表不是新知识点，而是把刚才代码的验收标准拆成可检查项。
```

```text
看图时先看 ______，再看 ______。如果你现在卡住，优先检查 ______。
```

## 二、不恶心读者（7 条）

### 7. 工具后置

先判断：读者不用这个工具能做这件事吗？能，就先讲怎么做，最后一句带过推荐工具。不能，才把工具放在步骤里。

**原则**：教程不是产品推广物料。工具名出现在正文里的唯一理由是该步骤必须用它。

### 8. 核心步骤全给

留半截等读者"加微信"或"买课"的——整篇删掉。要么全给，要么别写。

### 9. 禁用俯视语言

以下词汇和句式出现一次就重写：

- "众所周知"
- "显然"
- "大家都会"
- "你只需要"
- "很简单"
- "不过是"

替代方案：用"这一步做完你会看到……"代替"显然这一步很简单"。

### 10. 每篇至少一个"我在这栽过"

教程不是论文。AI 能输出完美内容，读者信的是那个掉过坑的人。

**格式**：Callout 高亮块，标题"我栽过的坑"。必须具体到：什么情况、什么后果、你怎么发现的。

### 11. 口语感 ≠ 语病

以下填缝词逐行删：

- "就是说""那个啥""对吧对吧""然后然后然后"
- "的话""嘛""啦""呀"（句尾语气词减半）

口语是风格，语病是敷衍。写完通读一遍，删掉所有不传递信息的词。

### 12. 晒成绩配资源透明

非要提自己的成果，后面必须跟一句资源披露：

> 我当时有 X 资源、Y 时间、Z 团队。你没有的话，可以这样替代：______

没有替代方案的炫成绩，删掉。

### 13. 不切篇，不埋收费伏笔

免费就全给，付费就标题写清楚。不在免费内容里插入"完整版请购买"。

## 三、结构完整（4 条）

### 14. 前五行 = 时间 / 目标 / 准备

教程正文前五行内，读者必须知道三件事：

```
⏱ 时间：约 X 分钟
🎯 学完能做什么：______
📋 需要准备：软件 X（版本 Y）/ 知识 Z
```

### 15. 前 200 字内动手

不许讲行业趋势、市场规模、"为什么要学这个"。读者点进来就是信你教的东西有用。

### 16. 无练习不教程

结尾必须有"今晚交作业"：

> 打开 X → 做 Y → 你会看到 Z → 截图发群里

练习要求：可在 5 分钟内完成 + 做完能立刻验证结果。

### 17. 前置阅读 + 下一步

每篇末尾两行链接：

```
← 前置阅读：[文章标题]
下一步：[文章标题] →
```

读者时刻知道自己在体系里哪个位置。

## 四、交付规范（5 条）

### 18. 开头必写环境

不是"打开 Photoshop"，是：

> Photoshop 2024 (25.0)，macOS Sequoia，2026 年 6 月

格式：软件名 + 版本号 + 操作系统 + 写作日期，四者缺一不可。

### 19. 静图 + 标注，不用动图

标注只用三种元素：红框、数字序号、箭头。

三种必须截图的时机：
1. 需要点某个按钮或菜单的时候
2. 操作前后的结果对比
3. 最容易卡住的界面（"你可能会看到这个，别慌"）

不截图的情况：纯文字概念解释不需要图。判断标准——读者不看这张图，操作会卡住吗？会就截，不会就不截。

### 20. 飞书排版四件套

- **Callout 块**（高亮卡片）：放"注意""提示""常见错误""我栽过的坑"
- **表格**：放对比（新手 vs 正确做法、方法 A vs 方法 B 适用场景）
- **代码块**：放操作命令、配置文本、代码片段
- **折叠块**：放扩展阅读、补充说明——正文保持短，想深入的人自己点开
- **画板类图**：放流程图、思维导图、对比图、层级图、决策树等核心结构图

### 21. 结构完整性

每篇教程三段式，缺一不可：

```
[开头] 时间 / 目标 / 准备 / 环境
  ——— 分割线 ———
[正文] 步骤 1 → 步骤 2 → 步骤 3（步骤间用分割线隔开）
  ——— 分割线 ———
[结尾] 练习 + 前置阅读 + 下一步
```

### 22. 场外文档引用

知识库中的源文档链接必须置顶或附在对应段落后：

```
📎 扩展阅读：[文档标题](链接)
```

链接文字要有标题，不放裸 URL。

### 22.1 外部资料要消化，不要堆砌

可以适度联网搜索，但搜索结果必须转化成教学内容。

**好**：根据官方文档，`docs +create` 创建的是可编辑云文档，所以本教程用它表达高亮块、表格和检查项。

**烂**：复制三段官方说明，然后继续写自己的步骤。

## 五、线下培训特供（4 条）

### 23. 零术语的"零"线上移

不需要解释"什么是 prompt""什么是 API"。但本教程专属的新概念仍需零术语——类比不能省。

有基础不代表全会。教新东西照样从例子切入。

### 24. 练习必须课堂可做答

每个练习满足三个条件：
- 5 分钟内能完成
- 做完能立刻验证结果（对/错/效果差异）
- 不需要额外下载或注册

不能在课上完成的练习不叫练习，叫作业。作业放课后。

### 25. 每个大段落后埋讨论节点

教程是讲师脚本，不是纯阅读材料。关键节点标注：

```
🗣 停下来讨论：你遇到过这种情况吗？你们是怎么解决的？
```

让教程从单向灌输变双向对话。每篇至少 1 个讨论节点。

### 26. 实操作业给现成的，带合格标准

特定章节如需要实操，提供完整的、可直接复制使用的材料和明确的通过条件：

```
🛠 实操作业

任务：用今天学的方法，把你项目里的日志系统加上错误追踪

用到的提示词（直接复制）：
"""
你是一个 Python 后端工程师。下面是一段日志代码，请帮我在不改变
原有逻辑的前提下，加上 sentry 错误追踪。只输出改动后的代码，
不要解释。

[粘贴你的日志代码]
"""

✅ 合格标准：
- [ ] 代码运行不报错
- [ ] 错误发生时能在 sentry 后台看到错误详情（截图）
- [ ] 原有日志功能不受影响（原来打印什么还打印什么）

🔍 自检：做完后对照合格标准逐项打勾，全勾了再提交。
提交方式：截图 sentry 后台 + 改动后的代码，发到群里
```

**烂**："请尝试在你自己项目中实现这个功能"——没材料、没标准，等于没布置。
