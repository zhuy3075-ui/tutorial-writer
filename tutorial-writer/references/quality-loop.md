# quality-loop.md — 轻量 Quality Loop

这份规范用于在正式产出前做有限轮质量检查。Loop 是质量守门，不是绕过用户确认的理由。

## 一、边界

- 不启用 L3 无人值守。
- 不自动改知识库。
- 不自动覆盖旧飞书文档。
- Loop 不能绕过需求确认和大纲审查。
- 用户未确认需求前，不进入大纲 Loop。
- 用户未确认大纲前，不进入正文、图文资产或飞书发布 Loop。

## 二、Loop 类型

| Loop | 触发时机 | 检查重点 | 默认上限 |
| --- | --- | --- | --- |
| 需求诊断 Loop | 输出项目级需求诊断结果前 | 9 个维度是否缺关键项；老师视角和产品经理视角是否都有判断 | 2 轮 |
| 大纲 Loop | 输出全流程规范大纲前 | 是否符合全流程规范；H1-H4、飞书块、图文、作业、风险是否齐全 | 2 轮 |
| 内容 Loop | 正文草稿完成后 | 是否可读、易懂、齐全；是否少废话、有过渡、有判断标准 | 2 轮 |
| 图文资产 Loop | 截图、画板、img2.0 规划和生成后 | 图是否服务教学；是否与章节上下文强绑定；数量是否匹配长度 | 2 轮 |
| 飞书发布 Loop | 创建或更新飞书文档后 | `/docx/` 链接、富文本块、图片插入、提示词文档、fetch 验证 | 3 轮 |

超过上限仍不通过时，停止并输出问题报告，等待用户判断。

## 三、各 Loop 检查项

### 需求诊断 Loop

- 项目类型是否明确。
- 是否有推荐教程形态和飞书承载方式。
- 是否覆盖项目目标、学员画像、课程目标、内容边界、素材来源、审美表达、图文互动、发布交付、风险验收。
- 是否体现老师视角：学习坡度、卡点、练习、复现能力。
- 是否体现产品经理视角：交付物、使用场景、发布范围、素材依赖、风险和验收。
- 是否把未确认项留给用户，而不是偷偷假设。

### 大纲 Loop

- 大纲是否来自已确认的诊断结果。
- H1/H2/H3/H4 是否层级正确。
- 每章是否有前言和过渡。
- 每节是否有知识点、飞书块、截图/图表、作业、验收标准。
- 图文资产规划是否服务教学，不是为了凑图。
- README、GitHub、部署、Wiki、Base/表单是否按需要规划。

### 内容 Loop

- 读起来是否像人在教，而不是模板拼接。
- 是否做到可读、易懂、齐全。
- 是否少用代码堆砌，多用自然语言解释判断。
- 每个知识点是否有“为什么、怎么做、常见错、5 分钟练习”。
- 代码块、表格、画板、图片、高亮块之间是否有承接说明。

### 图文资产 Loop

- 每张截图是否对应真实操作或真实界面。
- 每个红色标注是否指向准确，并解释标注内容和目的。
- 每个画板是否有清晰层级、节点关系和图类型。
- 每张 img2.0 图片是否绑定章节、知识点、上下文、教学问题和图下注释。
- 图片数量是否与教程长度匹配；不允许过少或过多。
- 自检不合格的图片要重写提示词重生成；不能把不相关图片插入教程。

### 飞书发布 Loop

- 已按场景使用飞书 CLI：首次发布用 `lark-cli docs +create --api-version v2`，修改已有文档用 `lark-cli docs +update --api-version v2`。
- 返回链接是 `/docx/`。
- 如果是修改已有文档，验证结果仍指向原 docx 链接或 token，没有意外新建链接。
- 已用 `docs +fetch --api-version v2` 验证标题、富文本块和图片存在。
- DocxXML 中有 callout、table、ul/ol、checkbox、grid/whiteboard/image/pre 等适用块。
- 如有 img2.0 图片，图片已插入正文对应位置。
- 已创建或交付独立生图提示词文档。

## 四、问题报告模板

```text
# Quality Loop 问题报告

Loop 类型：
已执行轮次：
仍未通过的问题：
影响：
推荐处理：
需要用户判断：
```

## 五、最终报告要求

最终报告说明：

- 哪些 Loop 已执行。
- 哪些检查通过。
- 哪些检查没有执行以及原因。
- 是否有用户确认需求和大纲。
- 是否有剩余风险。
