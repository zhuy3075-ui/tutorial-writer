# needs-diagnosis.md — 项目级需求诊断

这份规范用于把用户的模糊想法变成可审查的教程需求。诊断不是固定问卷，而是 agent 主导的项目判断。

## 一、先判断项目类型

模式选择后，先根据用户已给信息判断项目类型；信息不足时只问最关键的 1-2 个问题。

可选类型：

| 项目类型 | 适用场景 | 默认教程形态 |
| --- | --- | --- |
| 单篇教程 | 教一个明确动作或知识点 | 单篇飞书文档 |
| 小合集 | 2-5 个连续能力点 | 总览文档 + 小节文档 |
| Wiki 课程 | 长期维护的系统课程 | 飞书 Wiki / 课程文件夹 |
| 项目实战文 | 有成果展示、制作过程和复盘 | 公众号项目实战型长文 |
| 工具操作教程 | 教某个工具、插件或平台用法 | 操作指南 + 常见错误 |
| 部署交付教程 | 重点是 GitHub、README、部署、交付链接 | 交付流程文档 |
| 线下课讲义 | 面向课堂讲授和学员练习 | 讲师脚本 + 练习卡 |
| 参考文档复刻 | 学习某篇优质教程的结构和排版 | 参考模式迁移 |

如果已有信息足够，直接写：

```text
我的判断：这次更像是“____”，因为____。
推荐教程形态：____。
推荐飞书承载方式：____。
```

## 二、双角色诊断视角

诊断时同时扮演两个角色，不要只做材料摘要。

### 老师视角

必须判断：

- 学员现在会什么。
- 学员最容易卡在哪里。
- 本教程应该先讲场景、先展示成果，还是先讲方法。
- 哪些地方必须截图，哪些地方更适合画板、流程图、思维导图、对比图或层级图。
- 哪些练习能在 5 分钟内完成并验证。
- 哪些内容太难，需要拆成前置小节。
- 学完后学员能不能独立复现；如果不能，缺哪一个台阶。

### 产品经理视角

必须判断：

- 用户真正要交付什么，不要把“写教程”当成最终目标。
- 这份教程服务谁、在哪发布、怎么被使用。
- 成功标准是什么：看见什么、交付什么、学员能做什么。
- 哪些素材缺失会阻塞：截图、案例、知识库、链接、账号、API、README、GitHub。
- 哪些风险会影响发布：陈旧知识、权限、版权、隐私、工具版本、飞书能力限制。
- 是否需要 GitHub、README、部署、飞书 Wiki、Base、表单或任务。
- 交付物是否可维护、可复用、可继续扩展。

## 三、深度访谈模式

深度访谈用于高度个性化、用户目标模糊、学员画像不清、课程要服务具体人群、或用户主动要求“多聊聊再写”的场景。它不是默认流程；目标清楚的简单教程应走快速或标准诊断。

深度访谈的核心是：先从具体细节进入，再逐步扩展到整体目标。不要一上来问宏大问题，也不要一次性抛出长问卷。

### 访谈节奏

每轮只问 2-4 个问题。每轮结束后先复述理解，再判断是否还缺关键项。

```text
第 1 轮：具体场景
先问用户最近一次想写教程的具体情境：给谁看、为什么要写、现在卡在哪、希望对方看完能做什么。

第 2 轮：学员与使用方式
再问学员水平、常见误区、学习耐心、课堂还是自学、是老师讲还是学员自己看。

第 3 轮：表达偏好与审美
再问用户喜欢的阅读体验：公众号实战文、讲义、操作手册、Wiki、图多还是文字多、要不要轻松一点。

第 4 轮：交付和风险
最后问发布位置、是否要飞书 docx、是否已有原文档要更新、是否需要 README/GitHub/图片/表单/Base、哪些内容不能出错。
```

### 必须形成“用户画像与教程适配判断”

深度访谈后，在项目级需求诊断结果里补充：

```text
用户画像与教程适配判断：
- 用户真正想解决的不是：____
- 用户真正需要的是：____
- 用户适合的教程语气：____
- 用户适合的内容密度：____
- 用户适合的图文比例：____
- 用户最不能接受的写法：____
- 教程必须照顾的学习卡点：____
```

### 访谈边界

- 只了解与教程质量相关的信息：角色、经验、学习习惯、表达偏好、交付场景、风险边界。
- 不追问隐私、身份敏感信息、个人经历细节，除非用户主动提供且与教程直接相关。
- 用户已经表达清楚时，停止追问，给推荐方案。
- 用户犹豫时，给 2-3 个备选方案，让用户选择，而不是继续问散问题。

## 四、9 个诊断维度

需求诊断必须覆盖 9 个维度，但不要一次性把 30 个问题扔给用户。每轮只问最影响质量的 3-7 个问题。

| 维度 | 要判断什么 |
| --- | --- |
| 项目目标 | 最终要交付什么成果，为什么现在要做 |
| 学员画像 | 学员当前水平、已有经验、常见误区 |
| 课程目标 | 学完后能独立完成什么，课堂如何验证 |
| 内容边界 | 必讲、简讲、不讲、需要拆分的内容 |
| 素材来源 | 知识库、飞书链接、截图、代码、案例、README、GitHub |
| 审美表达 | 希望像操作手册、公众号实战文、课程讲义还是 Wiki 文档 |
| 图文互动 | 截图、画板、img2.0、表格、callout、checkbox、打卡 |
| 发布交付 | 单文档、多文档、Wiki、GitHub、README、部署链接、提示词文档 |
| 风险验收 | 陈旧知识、权限、隐私、版权、版本、飞书能力和验收标准 |

## 五、动态问题策略

### 第一层：识别项目类型

如果用户没说清楚，先问：

```text
你这次更像是：
1. 教一个具体操作
2. 做一篇项目实战文
3. 做一套课程/小合集
4. 把知识库整理成飞书教程
5. 复刻某个优秀教程风格
```

### 第二层：补关键缺口

按项目类型追问，不重复问用户已经给过的信息。

| 项目类型 | 优先追问 |
| --- | --- |
| 项目实战文 | 成品是什么、流程几步、素材有哪些、是否有截图/录屏、是否需要部署、复盘重点 |
| 工具操作教程 | 工具入口、版本、常见错误、哪些步骤必须截图、学员如何验收 |
| 课程合集 | 模块数量、课时长度、阶段作业、阶段验收、是否用 Wiki |
| 知识库整理 | 知识库范围、陈旧风险、核心知识点、是否允许联网校准 |
| 部署教程 | GitHub、README、目标平台、AI 工具、部署链接、交付检查 |
| 线下课讲义 | 课堂时长、学员设备、老师演示节奏、5 分钟练习、讨论点 |
| 参考文档复刻 | 参考链接、喜欢的结构、不要照搬的地方、可迁移的排版块 |

### 第三层：给推荐方案

如果用户回答模糊，不要继续追问散问题。给 2-3 个方案并明确推荐。

```text
我建议这份教程按“公众号项目实战型”做，因为你更关注结果展示、制作流程和复盘。

备选方案：
A. 单篇实战文：更快交付，适合分享。
B. 小合集：更系统，适合课程。
C. Wiki：适合长期维护，但制作成本更高。

我推荐 A，除非你准备做连续课程。
```

### 选项式推荐

当环境支持弹窗、选项卡、计划模式选项或 `request_user_input` 类工具时，把推荐方案做成可点击选项，不要求用户手动回复编号。每个选项写清楚适用场景和代价。

```text
推荐项：标准诊断
理由：信息还不够直接生成，但也没复杂到需要长访谈。

备选项：
- 快速生成：更快，但个性化较弱。
- 深度访谈：更贴合真实需求，但需要多轮对话。
```

## 七、停止条件

- 用户未确认“项目级需求诊断结果”前，不生成全流程规范大纲。
- 用户要求改目标、受众、范围、风格或交付物时，先更新诊断结果并再次等待确认。
- 只有用户明确说“继续 / 按这个做 / 你自行判断继续”，才进入 `outline-review.md`。
