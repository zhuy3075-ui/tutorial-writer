# content-depth.md — 教程内容深度与逻辑衔接规范

这份规范解决一个问题：教程不能像从知识库里摘句子拼起来。教程必须像讲师在带课：有主线、有过渡、有解释、有练习、有外部资料校准。

## 一、允许并建议适度联网搜索

当出现以下任一情况时，必须适度联网搜索：

- 主题是近两年快速变化的工具、平台、模型、框架、价格、政策或 API。
- 知识库材料只有观点，缺少定义、步骤、案例、限制或版本信息。
- 需要给小白解释一个行业新词、工具生态或最佳实践。
- 用户明确问“是否可以联网搜索相关资料”或要求补充外部资料。

联网搜索规则：

- 优先官方文档、产品文档、GitHub README、论文、标准文档、权威博客。
- 如果教程涉及具体项目，项目 README 和 GitHub 仓库是一级资料源；必须优先读取 README，再决定部署、运行和验收写法。
- 至少交叉验证 2 个来源；如果只找到 1 个可靠来源，明确说明。
- 外部资料只用来补足事实、定义、版本差异、案例和限制，不要把教程写成资料摘抄。
- 引用外部资料时，把“来源标题 + 链接 + 用途”放在教程开头或扩展阅读里。
- 如果外部资料和知识库冲突，指出冲突点，并优先采用更新、更权威的资料。
- 联网校准只服务教程内容，不自动改写知识库仓库。

### 陈旧知识校准流程

从知识库提取知识点后，先给每个知识点做风险标记：

| 风险类型 | 示例 | 是否需要联网 |
|----------|------|--------------|
| 稳定经验 | 学习路径、常见误区、个人案例 | 通常不需要 |
| 工具版本 | CLI 命令、软件入口、模型能力 | 需要 |
| 平台政策 | 权限、价格、限制、开放范围 | 需要 |
| API / SDK | 参数、scope、返回结构 | 需要 |
| 行业说法 | 新概念、最佳实践、趋势判断 | 建议 |

校准记录格式：

```text
知识点：______
知识库原说法：______
陈旧风险：低 / 中 / 高
联网确认：______
采用方式：保留 / 修正 / 删除 / 标注为历史经验
进入教程的位置：______
来源：______
```

如果知识库内容是“历史经验”，不要强行改成最新资料；可以写成：

```text
当时的做法是 ______。现在更推荐 ______，因为 ______。
```

不要联网的情况：

- 用户明确要求只使用本地知识库。
- 主题是用户私有经验复盘，外部资料会稀释原始经验。
- 已有材料足够完成教程，且事实不依赖最新版本。

## 二、先搭知识逻辑链，再写正文

正式写正文前，先产出一个“知识逻辑链”。它不一定展示给最终读者，但必须指导写作。

格式：

```text
读者当前状态：______
最终能做到：______
中间需要跨过的 3-5 个台阶：
1. 先理解 ______，否则会 ______
2. 再学会 ______，因为 ______
3. 然后处理 ______，这一步容易 ______
4. 最后验证 ______，判断标准是 ______
```

每个知识点之间必须有“为什么现在讲它”的过渡：

```text
上一节解决了 ______。
但做到这里，你会遇到 ______。
所以这一节要讲 ______。
```

禁止：

- 连续罗列知识点，没有解释先后顺序。
- 每节都像独立卡片，互相没有因果关系。
- 直接从概念跳到命令，中间不解释为什么要这么做。

### 主线复述检查

写正文前，必须能用一句话说清这篇课的主线：

```text
这篇课带你从 ______，经过 ______、______、______，最后能 ______。
```

如果这句话说不出来，说明大纲还只是材料列表，不是教程。

### 由浅入深的坡度

教程必须像一篇讲得好的图文长文：先让读者看到熟悉场景，再暴露问题，再给模型/方法，最后落到操作和判断。不要一上来讲框架、术语或命令。

推荐坡度：

```text
熟悉场景 → 小问题 → 为什么旧办法不够 → 新方法一句话 → 图解结构 → 自然语言操作 → 必要命令/配置 → 判断标准 → 小练习
```

每一层只解决一个读者问题：

| 层级 | 读者心里问什么 | 内容形态 |
|------|----------------|----------|
| 场景 | 这和我有什么关系 | 生活化例子 / 课堂场景 |
| 问题 | 我会在哪里卡住 | 失败表现 / 常见误区 |
| 方法 | 新办法到底变了什么 | 一句话模型 / 类比 |
| 图解 | 结构怎么一眼看懂 | 画板 / 图片 / 表格 |
| 操作 | 我具体怎么说、怎么点 | 自然语言 prompt / 少量命令 |
| 判断 | 怎么知道做对了 | 检查表 / 对比表 |

如果教程前 30% 已经出现大量代码、命令或配置，说明坡度太陡。先补场景、问题、类比和图解。

### 项目实战文章的主线

如果教程是在讲"我用 AI 工具做了一个完整项目"，主线必须从结果倒推过程：

```text
最终成品是什么 → 为什么想做它 → 我把想法拆成哪些部分 → AI 先帮我产出什么 → 我如何补充偏好 → 素材/内容/页面/部署如何推进 → 预览后怎么反馈 → 这个流程能迁移到哪里
```

这类文章的重点不是炫成品，而是让读者学会"如何把模糊想法交给 Agent 并持续推进"。

每个阶段都必须回答：

| 阶段问题 | 必须写清 |
|----------|----------|
| 这一阶段的输入是什么 | 想法、截图、文案、参考图、偏好、清单 |
| AI 做了什么 | 文档、题库、素材方案、页面、部署、修改 |
| 人做了什么判断 | 选方案、补偏好、确认题库、检查视觉、提出反馈 |
| 下一阶段为什么能开始 | 已经有了什么可复用产物 |

禁止把项目实战写成工具宣传。工具介绍必须从项目过程里长出来：先讲它在项目里承担了什么角色，再总结它适合什么任务。

### 拼凑检测红线

任意相邻两节调换顺序后，如果教程仍然成立，说明逻辑链失败。必须重排结构或重写过渡。

判断方法：

- 这一节是否依赖上一节的概念、结果或问题？
- 这一节是否解决上一节自然暴露的新问题？
- 下一节是否使用了这一节产出的判断、材料或能力？

三个问题都答不上来，说明这一节是“贴上去的”，不是“长出来的”。

### 块级逻辑桥检查

除了章节之间，块和块之间也要有因果关系。每个代码块、表格、画板、图片、Callout 都必须能说清：

```text
上一块交代了什么：______
下一块为什么现在出现：______
读者看完下一块要多一个什么判断：______
```

高风险硬切组合：

- 代码块 → 表格：中间必须解释“代码结果如何变成表格里的验收项”。
- 代码块 → 画板：中间必须解释“画板是在总结流程、角色关系，还是排错路径”。
- 表格 → Callout：中间必须解释“从表格中读出了哪个风险/结论”。
- 画板 → 步骤：中间必须解释“图里的哪一段对应接下来的操作”。
- 图片 → 正文：中间必须解释“图片缓解了什么理解压力，不是装饰”。

如果删掉两个块之间的桥接句，读者仍能懂，说明桥接句可能是废话；如果读者会问“为什么突然出现这个”，说明桥接句必须补。

## 三、解释深度最低标准

每个核心知识点至少写清楚 5 层：

1. **场景**：你在什么情况下会遇到它。
2. **问题**：不懂它会卡在哪里。
3. **原理**：用生活类比或简单模型解释它为什么这样工作。
4. **操作**：具体怎么做，每步能看到什么结果。
5. **判断**：怎么知道做对了，错了会有什么表现。

如果某一节只有“是什么 + 怎么做”，说明解释太薄，必须补“为什么”和“怎么判断”。

### 解释不足判定

出现以下情况，必须重写这一节：

- 只有定义，没有说明读者会在什么场景遇到它。
- 只有步骤，没有说明每一步解决什么问题。
- 只有命令，没有说明命令成功后应该看到什么。
- 只有经验判断，没有给读者可验证的标准。
- 只有外部资料摘录，没有转成教学语言。

## 四、过渡句模板

用过渡句把知识点串起来，不要硬切。

常用模板：

```text
到这里，你已经能 ______。下一步的问题是 ______。
```

```text
这一步看起来像是在配置工具，其实是在解决 ______。
```

```text
如果前一步没做好，这一步会出现 ______。所以我们先检查 ______。
```

```text
现在你知道了 ______，但真正上手时还会卡在 ______。
```

```text
前面讲的是“做出来”，这一节讲“怎么判断它做对了”。
```

## 五、外部资料如何融入教程

外部资料不能堆在正文里。每条外部资料必须服务于一个教学目的：

| 用途 | 放在哪里 | 写法 |
|------|----------|------|
| 纠正定义 | 概念第一次出现处 | “按官方文档，这里指的是……” |
| 补版本差异 | 注意/折叠块 | “如果你用的是旧版，入口可能叫……” |
| 补最佳实践 | 步骤前后 | “官方更推荐这种方式，因为……” |
| 补案例 | 例子/练习 | “类似场景可以参考……” |
| 补限制 | 常见问题 | “这个功能目前不适合……” |

每篇教程外部引用建议 2-5 条。超过 5 条时，放进“扩展阅读”折叠块，不要打断主线。

### 资料消化规则

每条外部资料必须转化为以下至少一种教学内容：

- 定义校准：帮助读者理解“它到底是什么”。
- 版本提醒：告诉读者不同版本入口、命令或限制不同。
- 操作依据：解释为什么这一步要这样做。
- 注意事项：提醒什么情况下会失败。
- 案例材料：让读者看到真实场景。
- FAQ：回答读者最可能卡住的问题。
- 练习素材：变成可复制的 prompt、命令或模板。

禁止把外部资料原文大段搬进正文。资料是原料，不是成品。

### 来源冲突处理

如果知识库材料和外部资料冲突：

1. 先判断时间：哪个更新。
2. 再判断权威性：官方文档优先于二手文章，项目 README 优先于转述。
3. 最后判断适用场景：本地经验可能适合特定课堂，官方文档适合通用规则。

正文里要说明采用哪个结论，以及为什么采用。

## 六、图表必须服务逻辑

不要只为了好看而加图。每张图必须回答一个具体问题：

| 读者问题 | 推荐图表 | 应表达的层次 |
|----------|----------|--------------|
| 我应该先做什么后做什么？ | 流程图 | 输入 → 处理 → 输出 → 验证 |
| 这些概念是什么关系？ | 思维导图 / 层级图 | 中心概念 → 分支 → 子概念 |
| 两种方式怎么选？ | 对比图 / 决策树 | 场景 → 选择 → 原因 |
| 人、AI、工具怎么协作？ | 架构图 / 泳道图 | 角色 → 动作 → 产物 |
| 学习路径怎么走？ | 时间线 / 阶梯图 | 入门 → 练习 → 实战 → 复盘 |
| 为什么会出错？ | 因果图 / 鱼骨图 | 表现 → 原因 → 修复 |

图表数量建议：

- 短教程：1-2 张。
- 中等教程：2-4 张。
- 长教程：每个大章节最多 1 张核心图，避免图太多抢正文。

图表后必须有 1 段解释，告诉读者“怎么看这张图”。不能图放完就结束。

每张图表必须绑定一个读者问题：

```text
这张图回答的问题：______
图中最重要的 3 个节点：______
读者看完后应该能判断：______
```

如果图表回答不了具体问题，删掉或改成表格。

## 七、反拼凑检查

写完后逐条检查：

- [ ] 每个知识点前后都有过渡。
- [ ] 每节都解释了“为什么现在讲它”。
- [ ] 每个核心知识点至少覆盖场景、问题、原理、操作、判断。
- [ ] 外部资料被消化成教学内容，而不是整段搬运。
- [ ] 图表承载了流程、关系、对比或判断，不是装饰。
- [ ] 读者从第一节走到最后一节，能力是逐步增长的。
- [ ] 任意相邻两节不能随意调换；如果能调换，说明主线不够强。
- [ ] 教程能用一句话复述“从哪里走到哪里”。
