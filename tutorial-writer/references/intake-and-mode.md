# intake-and-mode.md — 启动问答、模式选择与参考文档学习

这份规范解决教程开始前的两个问题：

1. 用户未必一开始就知道自己想要什么成果。
2. 用户可能会提供一篇优质飞书教程，希望学习它的结构和排版，形成新的教程模式。

正式写教程前必须先完成本文件的启动流程。除非用户明确说“不要问，直接按我给的要求写”，否则不要跳过。

## 零、强制确认闸门

这是阻塞式流程，不是建议。

1. 使用 skill 后，第一步必须让用户选择模式。
2. 如果环境支持弹窗、选项卡或 `request_user_input` 类工具，优先用真正的选项式提问。
3. 如果环境不支持弹窗，就发送编号选项，并要求用户回复编号或模式名。
4. 发送模式选项后必须停止本轮推进，等待用户回复；不要继续读取知识库、联网、写正文、生成图片、创建飞书文档或上传。
5. 用户选完模式后，再做问答和“初级需求汇总”。
6. 输出“初级需求汇总”后再次停止，等待用户确认“继续”或提出修改。
7. 只有用户确认继续，或明确写出“你自行判断继续”，才能进入正式教程流程。

即使用户已经给了主题、知识库路径和参考链接，也要先弹出模式选择；这些信息只能作为已知条件填入后续需求汇总。

## 一、先弹出模式选择

使用 skill 后，先用简短选项让用户选择工作模式。不要一上来写正文，也不要自动替用户选择模式。

建议话术：

```text
先选一个模式，我再继续：

1. 教程生成模式：我帮你从知识库/资料生成正式飞书教程。
2. 需求梳理模式：先不写正文，我通过问答帮你弄清目标成果、痛点和交付物。
3. 参考学习模式：你给我优质飞书教程链接，我学习它的结构、排版和图文节奏，沉淀成新模式。
4. 自定义模式：你说一个特殊目标，我按你的目标组织流程。
```

如果环境支持弹窗或选项式提问，优先使用弹出的选项；如果当前运行环境不支持，就用上面的编号列表让用户回复。

发送选项后，本轮只做等待，不继续执行：

```text
请回复 1 / 2 / 3 / 4，或直接说模式名。我会先整理你的初级需求汇总，确认后再开始生成教程。
```

### 模式执行规则

| 模式 | 下一步 |
|------|--------|
| 教程生成模式 | 先做需求问答，再进入课程范围判断和教程生成 |
| 需求梳理模式 | 只输出初级需求汇总，不写正式教程，除非用户确认继续 |
| 参考学习模式 | 先读取飞书链接，抽取结构和排版，再询问是否应用到本次教程 |
| 自定义模式 | 先复述用户目标，再补 3-5 个关键问题 |

## 二、通过问答总结真实需求

模式选择后，用 5 个以内的问题收集关键输入。问题要短，优先让用户容易回答。问题发出后也要等待用户回复，不要自问自答继续生成。

必问问题：

```text
1. 你希望最后得到什么成果？例如：单篇飞书教程 / 小合集 / Wiki 课程 / 公众号式项目实战文。
2. 这份教程要解决谁的什么痛点？例如：新手不会拆任务、会复制但不会改、不会判断 AI 输出质量。
3. 学员现在大概会什么？学完后你希望他们能独立做什么？
4. 你已有的材料有哪些？例如：知识库路径、飞书链接、截图、代码、案例、参考文章。
5. 你更喜欢哪种呈现方式？例如：操作指南、概念讲解、案例拆解、公众号项目实战、参考某篇文档的风格。
```

如果用户已经在原始请求里回答了其中一些，不要重复问。直接把已知信息填入汇总，只追问缺失且会影响产出的 1-3 项。

## 三、输出初级需求汇总

问答后先输出一份“初级需求汇总”，让用户快速知道自己想要什么、卡在哪里、最终要验收什么。

格式：

```text
初级需求汇总

目标成果：______
目标受众：______
当前痛点：______
学完后的能力：______
已有材料：______
参考风格：______
建议模式：______
建议承载方式：单篇飞书文档 / 多篇文档 / Wiki
核心章节初稿：
1. ______
2. ______
3. ______
需要进一步确认：
- ______
验收标准：
- ______
```

汇总后询问：

```text
这份理解对吗？你可以直接回复“继续”，也可以改目标成果、痛点、受众或参考风格。
```

只有用户确认继续，或用户明确要求“你自行判断继续”，才进入正式教程写作。用户只是给出补充信息时，先更新汇总并再次确认，不要直接进入生成。

## 四、参考飞书链接学习模式

当用户提供优质飞书教程链接，并要求学习结构和排版时，必须读取文档内容，而不是凭链接猜测。

使用：

```bash
lark-cli docs +fetch --api-version v2 --doc "<飞书文档链接>" --doc-format xml --scope full --json
```

读取前遵守 `lark-doc` skill 的要求：创建、读取、编辑飞书文档时必须使用 v2 API；需要时读取对应的 lark-doc reference。

### 抽取内容

读取后提炼以下信息：

```text
参考文档模式卡

模式名称：______
适合主题：______
开头方式：______
章节顺序：______
标题层级：______
图文节奏：______
常用飞书块：callout / table / grid / whiteboard / image / checkbox / code
图片使用方式：截图 / 画板 / img2.0 插图 / 信息图 / 对比图
过渡方式：______
练习与验收方式：______
可复用规则：
1. ______
2. ______
3. ______
不应照搬的地方：
- ______
```

### 沉淀为新模式

如果用户要求“形成一份新的模式”，把参考文档模式卡加入本次教程规划中，命名为“参考模式：____”。不要自动改写 skill 文件，除非用户明确要求更新 skill。

如果用户明确要求把该模式长期加入 skill，再按 skill 更新流程修改 `tutorial-structure.md` 或新增 reference，并同步安装版。

## 五、限制与隐私

- 不复制参考飞书文档的大段原文，只学习结构、排版、图文节奏和教学设计。
- 如果飞书链接无权限或读取失败，说明失败原因，并让用户导出内容或截图后再继续。
- 学习参考模式不是照抄风格；必须结合本次目标受众、痛点和交付物改写。
- 用户只想做需求梳理时，不创建飞书文档，不生成图片，不进入完整教程工作流。
