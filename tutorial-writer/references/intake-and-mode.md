# intake-and-mode.md — 启动问答、模式选择与参考文档学习

这份规范解决教程开始前的三个问题：

1. 用户未必一开始就知道自己想要什么成果。
2. 用户可能会提供一篇优质飞书教程，希望学习它的结构和排版，形成新的教程模式。
3. 用户可能不是要新建文档，而是要在已有飞书 docx 链接上继续修改。

正式写教程前必须先完成本文件的启动流程。除非用户明确说“跳过需求诊断和大纲审查，直接按我给的要求写”，否则不要跳过。

## 零、强制确认闸门

这是阻塞式流程，不是建议。

1. 使用 skill 后，第一步必须让用户选择模式。
2. 如果环境支持弹窗、选项卡、计划模式选项或 `request_user_input` 类工具，优先用真正的选项式提问；不要在可点击选项可用时只发纯文本编号。
3. 如果环境不支持弹窗，就发送编号选项，并要求用户回复编号或模式名。
4. 发送模式选项后必须停止本轮推进，等待用户回复；不要继续读取知识库、联网、写正文、生成图片、创建飞书文档或上传。
5. 用户选完模式后，进入 `needs-diagnosis.md` 的项目级需求诊断，而不是固定问卷。
6. 输出“项目级需求诊断结果”后再次停止，等待用户确认“继续”或提出修改。
7. 只有用户确认继续，或明确写出“你自行判断继续”，才能进入全流程规范大纲。
8. 用户确认大纲前，不生成正文、图片、飞书文档或上传媒体。
9. 如果用户给出已有飞书 docx 链接，或上下文中已有本次创建的 docx 链接，后续修改默认进入“修改已有飞书文档模式”，不要新建链接。

即使用户已经给了主题、知识库路径和参考链接，也要先弹出模式选择；这些信息只能作为已知条件填入后续项目级需求诊断结果。

## 一、先弹出模式选择

使用 skill 后，先用简短选项让用户选择工作模式。不要一上来写正文，也不要自动替用户选择模式。

### 选项式提问规则

优先使用类似计划模式的可点击选项。每次 2-5 个选项即可，推荐项放第一位，并在描述里说明为什么推荐。

选项必须包含：

- `label`：1-5 个词，方便点击。
- `description`：一句话说明适用场景和取舍。
- 推荐项：写在第一位，标签可带“推荐”。

如果只能发文本，再使用下面的话术。

建议话术：

```text
先选一个模式，我再继续：

1. 标准诊断模式（推荐）：先做项目级需求诊断和大纲审查，再生成飞书教程，适合大多数正式教程。
2. 快速生成模式：只补最关键缺口，适合目标清楚、材料完整、需要快交付的单篇教程。
3. 深度访谈模式：我像老师和产品经理一样多轮追问，从细节聊到整体，适合你还说不清真正需求或要高度个性化。
4. 参考学习模式：你给优质飞书链接，我学习结构、排版和图文节奏，再迁移到本次教程。
5. 修改已有飞书文档：你给已有 docx 链接，我在原链接上更新，不新建文档。
6. 自定义模式：你说一个特殊目标，我按你的目标组织流程。
```

如果环境支持弹窗或选项式提问，优先使用弹出的选项；如果当前运行环境不支持，就用上面的编号列表让用户回复。

发送选项后，本轮只做等待，不继续执行：

```text
请回复模式名，或点选一个模式。我会先做对应深度的需求诊断，确认后再出全流程规范大纲。
```

### 模式执行规则

| 模式 | 下一步 |
|------|--------|
| 标准诊断模式 | 先做项目级需求诊断，再进入大纲审查和教程生成 |
| 快速生成模式 | 只追问最关键 1-3 个缺口，仍需输出需求诊断结果和大纲，但篇幅更短 |
| 深度访谈模式 | 按 `needs-diagnosis.md` 的深度访谈流程多轮澄清，从具体场景聊到整体交付 |
| 需求梳理模式 | 只输出项目级需求诊断结果，不写正式教程，除非用户确认继续 |
| 参考学习模式 | 先读取飞书链接，抽取结构和排版，再纳入项目级需求诊断 |
| 修改已有飞书文档 | 先 fetch 原文档，再诊断修改目标，后续用 `docs +update` 更新原链接 |
| 自定义模式 | 先复述用户目标，再按项目类型补关键缺口 |

## 二、通过诊断总结真实需求

模式选择后，读取 `needs-diagnosis.md`，先判断项目类型，再动态追问关键缺口。问题要短，优先让用户容易回答。问题发出后也要等待用户回复，不要自问自答继续生成。

不要固定按顺序问 30 个问题，也不要只保留旧式 5 问。诊断必须覆盖 9 个维度，但每轮只问最关键的 3-7 个问题：

```text
项目目标 / 学员画像 / 课程目标 / 内容边界 / 素材来源 / 审美表达 / 图文互动 / 发布交付 / 风险验收
```

如果用户已经在原始请求里回答了其中一些，不要重复问。直接把已知信息填入诊断结果，只追问缺失且会影响产出的 1-3 项。

### 推荐提问方式

先判断项目类型：

```text
你这次更像是：
1. 教一个具体操作
2. 做一篇项目实战文
3. 做一套课程/小合集
4. 把知识库整理成飞书教程
5. 复刻某个优秀教程风格
```

再按项目类型补关键缺口：

- 项目实战文：问成品、流程、素材、截图、部署、复盘。
- 工具操作教程：问操作入口、常见错误、截图、版本、验收。
- 课程合集：问模块、课时、作业、阶段验收、Wiki。
- 知识库整理：问知识库范围、陈旧风险、核心知识点、是否联网。
- 部署教程：问 GitHub、README、目标平台、AI 工具、验收链接。

如果用户回答模糊，给 2-3 个推荐方案并明确推荐，不要继续零散追问。

### 方案卡规则

涉及范围、风格、承载方式、是否深度访谈、是否更新原文档时，优先给方案卡而不是开放式追问。

格式：

```text
我推荐：____
推荐理由：____

备选方案：
A. ____：适合____，代价是____。
B. ____：适合____，代价是____。
C. ____：适合____，代价是____。
```

如果环境支持可点击选项，把 A/B/C 做成选项；如果不支持，再让用户回复方案名。

## 三、输出项目级需求诊断结果

问答后先输出一份“项目级需求诊断结果”，让用户快速知道自己想要什么、卡在哪里、最终要验收什么。

验收标准：

## 4. 内容边界

汇总后询问：

```text
这份诊断对吗？你可以直接回复“继续”，也可以改目标成果、学员水平、审美方向、交付方式或风险边界。
```

只有用户确认继续，或用户明确要求“你自行判断继续”，才进入全流程规范大纲。用户只是给出补充信息时，先更新诊断结果并再次确认，不要直接进入生成。

## 四、参考飞书链接学习模式

当用户提供优质飞书教程链接，并要求学习结构和排版时，必须读取文档内容，而不是凭链接猜测。

使用：

```bash
lark-cli docs +fetch --api-version v2 --doc "<飞书文档链接>" --doc-format xml --scope full --json
```

读取前遵守 `lark-doc` skill 的要求：创建、读取、编辑飞书文档时必须使用 v2 API；需要时读取对应的 lark-doc reference。

### 抽取内容

读取后提炼以下信息：

```text
参考文档模式卡

模式名称：______
适合主题：______
开头方式：______
章节顺序：______
标题层级：______
图文节奏：______
常用飞书块：callout / table / grid / whiteboard / image / checkbox / code
图片使用方式：截图 / 画板 / img2.0 插图 / 信息图 / 对比图
过渡方式：______
练习与验收方式：______
可复用规则：
1. ______
2. ______
3. ______
不应照搬的地方：
- ______
```

### 沉淀为新模式

如果用户要求“形成一份新的模式”，把参考文档模式卡加入本次项目级需求诊断中，命名为“参考模式：____”。不要自动改写 skill 文件，除非用户明确要求更新 skill。

如果用户明确要求把该模式长期加入 skill，再按 skill 更新流程修改 `tutorial-structure.md` 或新增 reference，并同步安装版。

## 五、修改已有飞书文档模式

当用户提供已有飞书 docx 链接，或说“在刚刚那个文档上改 / 不要新建 / 原链接修改 / 继续优化”，默认修改原文档。

执行规则：

1. 先用 `lark-cli docs +fetch --api-version v2 --doc "<飞书文档链接>" --doc-format xml --detail with-ids --json` 读取原文档，必要时只抓取相关章节。
2. 先输出“修改目标诊断”：哪些内容要改、哪些内容保留、是否会影响图片/画板/评论。
3. 用户确认后，用 `lark-cli docs +update --api-version v2 --doc "<飞书文档链接>"` 更新原文档。
4. 优先用 `block_replace`、`block_insert_after`、`str_replace` 做精确修改；只有用户明确要求整体重写时，才考虑 `overwrite`。
5. 修改后必须再次 `docs +fetch --api-version v2` 验证，最终返回同一个 `/docx/` 链接。

禁止行为：

- 用户只是说“修改 / 优化 / 调整”，就创建新飞书文档。
- 原文档里有图片、画板、评论时，直接 `overwrite` 导致不可恢复内容丢失。
- 没有 fetch 原文档就凭记忆改。

## 六、限制与隐私

- 不复制参考飞书文档的大段原文，只学习结构、排版、图文节奏和教学设计。
- 如果飞书链接无权限或读取失败，说明失败原因，并让用户导出内容或截图后再继续。
- 学习参考模式不是照抄风格；必须结合本次目标受众、痛点和交付物改写。
- 用户只想做需求梳理时，不创建飞书文档，不生成图片，不进入完整教程工作流。
