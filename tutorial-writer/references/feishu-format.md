# feishu-format.md — 飞书云文档排版规范

## 输出形态硬规则

默认输出是**已创建成功的飞书云文档（docx）**，不是本地草稿，也不是 Drive 中的普通 Markdown 文件。

但“飞书云文档”不等于所有内容都放进一篇文档。先根据 `course-planning.md` 判断是单文档、多文档，还是 Wiki 知识库结构。

当用户说“使用飞书 CLI”“上传到飞书文档”“做成飞书教程”时，必须实际执行：

```bash
lark-cli docs +create --api-version v2 --as user --content @./tutorial.xml --json
```

仅当用户明确说“上传 Markdown 文件 / 原生 .md 文件 / Drive 普通文件”时，才使用：

```bash
lark-cli markdown +create --as user --file ./tutorial.md
```

如果需要从本地 `.md`、`.docx`、`.txt`、`.html` 转成在线文档，也可以使用：

```bash
lark-cli drive +import --type docx --as user --file ./tutorial.md --json
```

教程正文必须优先使用 `docs +create` + DocxXML，因为它能稳定表达高亮块、表格、代码块、检查项、分栏和画板。不要为了省事用 `--doc-format markdown` 上传正文；Markdown 导入通常无法呈现足够好看的高亮块和富文本结构。

验收时必须确认返回链接包含 `/docx/`。如果返回 `/file/` 或只有 `.md` 文件 token，说明上传形态错了。

如果教程规划了 img2.0 生成图，先在正文放明确插图位，再生成图片，按 `image-illustration.md` 完成自检；自检合格后再用 `docs +media-insert` 把图片插入到对应位置附近。图片提示词文档也要用 `docs +create` 创建成单独飞书云文档，或作为明确附件交付。

### 禁止“只给命令不执行”

除非用户明确要求只要草稿，否则不能停在“这里是上传命令”。必须执行：

1. `lark-cli docs +create --api-version v2 ...`
2. `lark-cli docs +fetch --api-version v2 ...` 验证
3. 如有图片，执行 `lark-cli docs +media-insert ...`
4. 返回飞书文档链接

## 单文档 / 多文档 / Wiki 输出

### 单文档

适合单节教程。结构是：

```text
标题
开头高亮块
核心画板图
正文步骤
练习
作业
扩展阅读
```

### 多文档小合集

适合 2-5 节连续教程。至少产出：

- 1 篇总览文档：学习路径、章节目录、每节产出、作业说明。
- 多篇小节文档：每篇小节都能独立学习，但必须承接上一节。

总览文档里必须有章节表：

```xml
<table>
  <thead>
    <tr><th background-color="light-gray">小节</th><th background-color="light-gray">解决的问题</th><th background-color="light-gray">产出</th><th background-color="light-gray">下一节衔接</th></tr>
  </thead>
  <tbody>
    <tr><td>01</td><td>____</td><td>____</td><td>____</td></tr>
  </tbody>
</table>
```

### Wiki 知识库

适合完整课程或需要长期维护的训练营。先输出 Wiki 结构规划；只有用户确认 Wiki 空间和权限后，才创建 Wiki 节点。

Wiki 首页必须包含：

- 课程目标
- 学习路径图
- 模块目录
- 作业与验收
- 素材与扩展阅读
- 版本/更新说明

## 排版四件套

每篇教程必须使用以下四种飞书元素。同一种格式从头到尾的教程视为不合格。

### 0. 图文节奏

教程排版参考高质量公众号长文的阅读体验：短段落、明确小标题、图文交替、重点块突出。不是把 Markdown 转成飞书，也不是把代码和表格堆在一起。

推荐节奏：

```text
自然语言场景段
→ 高亮块总结本节要解决的问题
→ 图片/画板/表格帮助一眼看懂
→ 读图说明
→ 必要操作或 prompt
→ 检查表/对比表
→ 小结或下一节过渡
```

硬性要求：

- 连续两个代码块之间必须有自然语言解释或检查项，不允许代码连贴。
- 前 30% 内容应以场景、问题、图解为主，不要过早堆命令。
- 每 500-800 字至少出现一个视觉断点：callout、table、grid、whiteboard、image、checkbox 任选其一。
- 图不是装饰。图前说明为什么看，图后说明怎么看、怎么用。
- 代码块数量应少于自然语言解释块；能用 prompt 描述的，不用脚本表达。

#### 公众号项目实战排版

项目实战型教程必须先有"成果展示区"，再进入流程拆解。

推荐排版：

```text
开场短段
→ 最终效果图片/录屏占位
→ 完成点清单 callout
→ 项目流程画板
→ 分阶段小标题
→ 每阶段：说明段 + 截图/表格/清单 + 读图说明 + 下一阶段过渡
→ 坑点 callout
→ 工具能力表
→ 可迁移场景表
→ 小练习
```

成果展示区必须包含：

- 1 段说明最终成品是什么。
- 1 个图片、录屏占位或截图组。
- 1 个完成点清单，列出 3-6 个可见能力。

阶段拆解区必须避免"图片图片图片"式堆图。连续两张图之间必须有解释，说明前一张图证明什么、后一张图推进到哪里。

### 1. Callout 块（高亮卡片）

飞书文档中插入"高亮块"。用于以下内容：

| 类型 | 标题 | 示例 |
|------|------|------|
| 提示 | 💡 提示 | 补充信息，不影响主流程 |
| 注意 | ⚠️ 注意 | 重要但非致命的提醒 |
| 常见错误 | ❌ 常见错误 | 你踩过的坑 |
| 我栽过的坑 | 🕳 我栽过的坑 | 个人经验，必须具体到时间/场景/后果 |

Callout 块不放正文步骤。正文应脱离 Callout 仍能独立阅读。

### 2. 表格

用于对比和映射关系：

```
| 新手常犯 | 正确做法 | 为什么 |
|----------|---------|--------|
| ...      | ...     | ...    |
```

```
| 方法 A | 方法 B | 适用场景 |
|--------|--------|---------|
| ...    | ...    | ...     |
```

表格不超过 4 列。超过用折叠块 + 分段表格。

### 2.1 互动作业与学习打卡

作业、练习、章节参与确认不应只是普通文字。优先使用 `participation.md` 中的互动作业卡和学习打卡组件。

DocxXML 基础形态：

```xml
<callout emoji="📝" background-color="light-blue" border-color="blue">
  <p><b>本节作业</b></p>
  <checkbox done="false">我已经完成本节阅读</checkbox>
  <checkbox done="false">我完成了 5 分钟练习</checkbox>
  <checkbox done="false">我把结果截图发到群里</checkbox>
</callout>
```

当前 DocxXML 支持 `<button>`，但只适合打开链接、复制页面或关注页面；不要把它当成会自动记录点击人的打卡按钮。文档内 checkbox 只能作为学习引导或共享检查项；同一份文档里，一个人勾选后其他人也会看到变化。如果需要展示真实参与进度或头像墙，接入 Base / 表格 / 表单，并在文档里放入口按钮和占位区。单人学习场景可以提供 `DuplicatePage` 按钮，让学员复制个人副本后再勾选：

```xml
<callout emoji="✅" background-color="light-green" border-color="green">
  <p><b>我已参与</b></p>
  <checkbox done="false">我已完成本节学习和练习。</checkbox>
  <p><button action="OpenLink" src="https://example.com">打开打卡表</button></p>
  <p><button action="DuplicatePage" src="https://example.com">复制个人学习副本</button></p>
  <p>当前进度：由讲师在 Base / 表格中统计。</p>
  <p>[参与头像区占位：接入 Base 人员字段后展示已参与学员名单或头像]</p>
</callout>
```

不要伪造头像、名单或参与人数。无法自动统计时，明确标注为占位；已有 Base / 表格数据源时，才显示真实名单、头像或人数。

### 3. 代码块

以下内容必须放入代码块，不混在正文段落中：

- 终端命令
- 代码片段
- 配置文件内容
- prompt 原文

代码块必须标注语言类型（bash / python / json / yaml 等）。

代码块后不能直接接表格、画板、图片或 Callout。必须先用普通段落说明：

- 这段代码/命令刚刚完成了什么。
- 下一块视觉内容要帮助读者判断什么。
- 如果输出不符合预期，优先看哪里。

DocxXML 示例：

```xml
<p>先跑这条命令，确认你创建的是可编辑云文档，而不是普通 Markdown 文件。</p>
<pre lang="bash" caption="创建飞书云文档"><code>lark-cli docs +create --api-version v2 --as user --content @./tutorial.xml --json</code></pre>
<p>命令成功只代表文档创建出来了。接下来这张表用来判断它是不是“教学可用”：既要有 docx 链接，也要能看到高亮块、表格和代码块。</p>
<table>
  <thead>
    <tr><th background-color="light-gray">检查项</th><th background-color="light-gray">合格标准</th><th background-color="light-gray">不合格先改哪里</th></tr>
  </thead>
  <tbody>
    <tr><td>链接形态</td><td>包含 /docx/</td><td>改用 docs +create</td></tr>
    <tr><td>富文本块</td><td>能看到 callout/table/pre</td><td>补 DocxXML 块</td></tr>
  </tbody>
</table>
```

### 4. 折叠块

用于降低正文密度。以下内容放入折叠块：

- 扩展阅读
- 补充说明（不影响主流程的额外信息）
- 术语详解（想深入了解的人点开）
- 版本差异说明（"如果你用的是旧版本……"）

折叠块标题用问句或命令式，不用名词：
- **好**："为什么这里用 POST 而不是 GET？"
- **烂**："HTTP 方法说明"

### 5. 画板类图（必须）

每篇教程至少规划 1 个画板类图。画板用于承载**核心流程、思维结构、层级关系、对比结构、闭环、决策路径**，不是装饰图。

优先使用飞书 DocxXML 的画板块：

```xml
<whiteboard type="mermaid">
graph TD
  A[说清楚目标] --> B[AI 生成内容]
  B --> C[飞书 CLI 创建文档]
  C --> D[用户检查结果]
  D --> A
</whiteboard>
```

如果 Mermaid 不适合表达视觉层次，使用 SVG 画板：

```xml
<whiteboard type="svg">
  <svg viewBox="0 0 1200 720" xmlns="http://www.w3.org/2000/svg">
    <!-- 自包含 SVG，不引用外部图片、字体、脚本 -->
  </svg>
</whiteboard>
```

#### 画板选型

| 教程内容 | 推荐画板 | 用途 |
|----------|----------|------|
| 多步骤操作 | 流程图 / 泳道图 | 让小白知道先后顺序 |
| 概念关系 | 思维导图 / 层级图 | 展示中心概念、分支概念、子概念 |
| 方法对比 | 左右对比图 / 对比表 | 看清错误做法 vs 正确做法 |
| 工具协作 | 架构图 | 展示人、AI、CLI、飞书之间的关系 |
| 学习路径 | 时间线 / 阶梯图 | 展示从入门到实战的进阶 |
| 反馈循环 | 闭环图 / 飞轮图 | 展示“生成-验证-修正”的循环 |
| 判断选择 | 决策树 | 展示不同场景下怎么选 |
| 错误排查 | 因果图 / 鱼骨图 | 展示问题表现、原因、修复 |

#### 各类图的结构标准

| 图表类型 | 必须包含 | 禁止 |
|----------|----------|------|
| 流程图 | 开始、3-7 个关键步骤、验证节点；箭头方向统一 | 节点只写名词、没有结果判断 |
| 思维导图 | 1 个中心节点、3-5 个一级分支、最多 2 层 | 拿思维导图表达操作顺序 |
| 对比图 | 左右或上下分区、差异点对齐、推荐项统一高亮 | 只靠 emoji 或加粗表达重点 |
| 层级图 | root → level 1 → level 2，不跳级；同层样式一致 | 父子关系混乱、同层大小不一 |
| 泳道图 | 每条泳道 1 个角色；跨泳道箭头表示交付物 | 一个泳道塞多个角色 |
| 决策树 | 分叉是可回答的问题；叶子节点是行动建议 | 分叉写抽象概念，叶子不给动作 |
| 因果图 | 表现 → 原因 → 修复；原因分组清楚 | 只罗列原因，不给修复方向 |

#### 画板美观标准

- 标题清楚：图上必须有一句图名，不只写节点。
- 层级分明：主流程 3-6 个主节点；子节点不超过 2 层。
- 视觉对齐：同级节点等宽、等高、等间距。
- 颜色克制：主色不超过 3 个，颜色语义固定。
- 文案短：单个节点不超过 14 个汉字；长说明放图下正文。
- 有方向：流程图必须有箭头；对比图必须有左右/上下分区。
- 不拥挤：节点之间留白明显，避免文字贴边。

#### 画板反模式

- 只放一个大标题，没有信息结构。
- 把整段正文塞进节点。
- 节点太多，读者不知道从哪里看。
- 颜色很多但没有语义。
- 图和正文重复，不能帮助理解。

#### 图表组合规则

- 操作指南型教程：至少 1 张流程图；如果有角色协作，再加泳道图。
- 概念讲解型教程：至少 1 张思维导图或层级图；如果有误区，再加对比图。
- 案例拆解型教程：至少 1 张问题-尝试-结果流程图；如果有复用场景，再加对比表。
- 工具/API 类教程：优先使用架构图 + 流程图，说明“谁调用谁、输入输出是什么”。
- 每张图后必须有 1 段“怎么看这张图”的解释，不允许放完图直接进入下一节。
- 每张图前必须有 1 句“为什么现在看这张图”；图不能突然插在代码或表格后面。

#### 图后解释模板

每张画板后必须跟一段解释：

```text
怎么看这张图：先看 ______，再看 ______，最后用 ______ 判断自己是否走对。
这张图和本节操作的关系是：______
```

### 6. img2.0 生成图片（按需）

img2.0 图片用于缓解学习压力、知识点演示和阶段汇总。它和截图、画板分工不同：

- 截图：真实界面位置。
- 画板：流程、架构、层级、决策。
- img2.0 图片：类比、场景化理解、知识点总结、一眼看懂的学习卡。

正文占位：

```xml
<p>[插图位 IMG-01：这里插入“AI 助教把复杂任务拆成小步骤”的教学插画。作用：缓解学习压力，并引出下一节的小步验证。]</p>
```

插入命令：

```bash
lark-cli docs +media-insert --api-version v2 --doc "<doc-url>" --file ./assets/images/IMG-01.png
```

提示词文档必须创建为飞书云文档，除非用户明确不需要：

```bash
lark-cli docs +create --api-version v2 --as user --doc-format markdown --content @./assets/prompts/tutorial-img2-prompts.md --json
```

验收：

- 图片靠近它服务的段落。
- 图片下方或前后有一句解释“这张图怎么看”。
- 图片没有替代截图或画板的职责。
- 图片提示词文档可追溯每张图片的生成目的和提示词。
- 如果规划了“需要生成”的 img2.0 图片，不能只留下插图位；必须实际生成、自检合格并插入。

## 结构布局

```
┌─────────────────────────────┐
│ 标题（H1）                   │
├─────────────────────────────┤
│ 开头高亮块                   │
│ ⏱ 时间 / 🎯 目标 / 📋 准备   │
│ 🖥 环境：XX v1.0, macOS, 日期 │
├─────────────────────────────┤
│ 核心画板图                   │
│   流程 / 架构 / 对比 / 闭环   │
├─────────────────────────────┤
│ 正文 H2                      │
│   步骤 1 H3                  │
│     正文 + [配图位]          │
│   ——— 分割线 ———            │
│   步骤 2 H3                  │
│     正文 + [配图位]          │
│   ——— 分割线 ———            │
│   步骤 3 H3                  │
│     正文 + [配图位]          │
├─────────────────────────────┤
│ 🗣 停下来讨论                 │
├─────────────────────────────┤
│ 📝 练习                      │
├─────────────────────────────┤
│ ← 前置阅读：[标题]            │
│ 下一步：[标题] →              │
└─────────────────────────────┘
```

### 分隔线使用规则

- 步骤之间：必须用分割线隔开
- 开头和正文之间：分割线
- 正文和结尾之间：分割线
- 同一步骤内的小节：不用分割线，用 H3/H4 区分

### 标题层级

- H1：教程标题（一篇只有一个）
- H2：大段落（"为什么需要""怎么做""常见错误"）
- H3：步骤编号（"步骤 1：安装环境"）
- H4：子步骤（少见，超过 3 个 H4 考虑拆成两步）

### 标题与序号层级

标题和序号必须能在飞书中正确显示层级，不能靠空格伪造层级。

#### 标题规则

- 一篇只有一个 `<title>`，它对应飞书文档标题。
- 正文顶层章节用 `<h1>`；章节内步骤用 `<h2>`；步骤内小节用 `<h3>`；必要时才用 `<h4>`。
- 不跳级：不能从 `<h1>` 直接跳到 `<h3>`。
- `<h2>` 表示大章节时不要写“步骤 1”；如果教程是操作指南，步骤编号放在 `<h2>` 也可以，但全篇必须一致。
- 同级标题句式保持一致，例如都写“步骤 1：____”或都写动宾短语。
- 标题承担导航，不写成完整长段落；超过 24 个汉字就拆短。
- 连续超过 3 个 `<h4>`，说明这一节太碎，应拆成新的 `<h3>` 或 `<h2>`。

#### 序号规则

- 大步骤用标题编号：`步骤 1：准备环境`。
- 步骤内连续动作使用 `<ol><li seq="auto">...</li></ol>`。
- 无先后顺序的要点使用 `<ul>`，不要强行编号。
- 检查清单使用 `<checkbox>`，不要用普通列表冒充。
- 子步骤最多两层；超过两层说明内容太复杂，应拆成新步骤。
- 有序列表只放同一层级的连续操作，不混入解释段。解释段放在列表前后。
- 截图数字序号必须与正文步骤编号一致。

DocxXML 示例：

```xml
<h1>怎么做</h1>
<h2>步骤 1：准备环境</h2>
<ol>
  <li seq="auto">打开终端。</li>
  <li seq="auto">执行版本检查命令。</li>
</ol>
<h3>如果版本不对怎么办</h3>
<p>先升级工具，再回到上一步验证。</p>
```

### 对比内容显著显示

对比、取舍、错误 vs 正确、前后变化，必须显著呈现，不能藏在段落里。

优先级：

1. 两种做法对比：用 `<grid>` 左右分栏。
2. 三个以上维度对比：用 `<table>`。
3. 错误风险：用红色或黄色 `<callout>`。
4. 关键结论：用绿色或蓝色 `<callout>`。
5. 复杂选择路径：用决策树画板。

DocxXML 示例：

```xml
<grid>
  <column width-ratio="0.5">
    <callout emoji="❌" background-color="light-red" border-color="red" text-color="red">
      <p><b>新手做法：</b>直接让 AI 写完整项目。</p>
      <p>结果：出错后不知道从哪里修。</p>
    </callout>
  </column>
  <column width-ratio="0.5">
    <callout emoji="✅" background-color="light-green" border-color="green" text-color="green">
      <p><b>推荐做法：</b>先做最小功能，再逐步验证。</p>
      <p>结果：每一步都能定位问题。</p>
    </callout>
  </column>
</grid>
```

## DocxXML 输出规范

生成飞书云文档时，优先输出 DocxXML。常用结构如下：

```xml
<title>教程标题</title>

<callout emoji="✅" background-color="light-green" border-color="green" text-color="green">
  <p>开头先给结论：这篇教程完成什么、适合谁、最终产出是什么。</p>
</callout>

<table>
  <thead>
    <tr><th background-color="light-gray">项目</th><th background-color="light-gray">内容</th></tr>
  </thead>
  <tbody>
    <tr><td>时间</td><td>约 25 分钟</td></tr>
  </tbody>
</table>

<whiteboard type="mermaid">
graph TD
  A[目标] --> B[步骤]
  B --> C[验证]
</whiteboard>

<pre lang="bash" caption="创建飞书云文档"><code>lark-cli docs +create --api-version v2 --as user --content @./tutorial.xml --json</code></pre>

<checkbox done="false">检查飞书链接是否包含 /docx/</checkbox>
```

### DocxXML 注意事项

- 标签本身不要转义；只转义文本里的 `<`、`>`、`&`。
- 代码块用 `<pre lang="..." caption="..."><code>...</code></pre>`。
- 重点提示用 `<callout>`，不要用普通加粗冒充。
- 检查清单用 `<checkbox>`，不要用普通无序列表冒充。
- 章节之间用 `<hr/>` 拉开视觉节奏。
- 表格表头使用 `background-color="light-gray"`。
- 如果文档较长，先 `docs +create` 创建骨架，再用 `docs +update` 分段追加。
- 标题用真实 `<h1>` / `<h2>` / `<h3>` 标签，序号用 `<ol>`，不要用纯文本空格模拟层级。
- 对比内容必须用 `<grid>`、`<table>`、`<callout>` 或 `<whiteboard>` 显著呈现。
- 只使用已被 `lark-cli docs +create --api-version v2` 支持的标签。
- Mermaid 画板优先使用兼容性高的 `graph TD` / `graph LR`。`mindmap`、复杂 `subgraph`、样式语法不确定时，改用 SVG whiteboard。
- SVG 必须自包含，不引用外部字体、图片、脚本。
- 如果 `<whiteboard>` 渲染失败，必须降级为表格/代码块中的画板插入位说明，不能静默省略。

### 视觉质量最低标准

飞书文档不能只是 Markdown 转换后的普通长文。至少满足：

- 开头必须有 `<callout>` 总结目标和最终产出。
- 前 1/3 必须出现一个结构化视觉块：表格、分栏、画板或图片。
- 每个核心章节至少有一种非纯文本块：callout / table / pre / checkbox / grid / whiteboard / image。
- 常见错误必须用警告或错误 callout，不埋在普通段落里。
- 对比内容必须显著显示，不只靠正文描述。
- 练习和验收必须用 `<checkbox>` 或表格呈现。
- 代码、命令、prompt 必须用 `<pre>`，不能混在段落里。

## 场外文档引用

知识库中的源文档链接置顶或附在对应段落下方。格式：

```
📎 相关文档：[文档标题](链接)
```

- 链接文字必须是文档标题，不放裸 URL
- 置顶引用放教程开头卡片下方
- 段落引用放对应知识点下方
- 引用不超过 3 条（超过用折叠块收纳）

## 排版反模式

以下写法禁止：

- 纯文字无任何排版元素的整篇教程
- 用"加粗"代替 Callout 块放重要提示
- 代码混在正文段落里不单独成块
- 表格超过 4 列
- GIF 动图
- 表情符号作为唯一视觉元素（不支持纯 emoji 标题）
- 用户要求飞书文档时，却用 `markdown +create` 上传普通 `.md` 文件
- 没有画板类图，或画板图只是装饰、没有承载核心信息
- 画板节点密密麻麻、层次混乱、颜色没有语义
- 标题层级跳跃，或用空格、数字手写伪造层级
- 对比内容只写在普通段落里，没有显著视觉结构

## 发布验证

用飞书 CLI 创建后，必须做最小验证：

```bash
lark-cli docs +fetch --api-version v2 --doc "https://xxx.feishu.cn/docx/..." --doc-format xml --scope full --json
```

检查：

- 返回 `ok: true`
- 链接是 `/docx/`
- XML 中能看到 `<title>`
- XML 中能看到 `<callout>`、`<table>`、`<pre>`、`<checkbox>`
- XML 中能看到 `<whiteboard>`，或至少在文档中明确标注了要插入的画板位
- XML 中标题层级合理，没有从 `<h1>` 跳到 `<h3>`
- 有序步骤使用 `<ol>` 或清晰标题编号，对比内容使用显著结构块
- 如果规划了 img2.0 图片，飞书文档中能看到 `<img>`，或 `docs +media-insert` 返回成功的 `file_token`
- 如果生成了图片提示词文档，也必须返回该飞书云文档链接
