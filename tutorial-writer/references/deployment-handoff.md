# deployment-handoff.md — 项目部署交付与 README 接入规范

这份规范用于教程里的项目部署、上线、交付和复刻环节。

## 一、部署环节写给谁

默认不是让学员手写部署脚本，而是把部署任务交给 AI 编程工具：

- Codex
- WorkBuddy
- Claude Code

教程要教会学员如何用自然语言把项目、目标、约束和验收标准交给这些工具。

不要把部署环节写成代码细节讲解。除非用户明确要求，不展开解释配置文件、框架源码或构建脚本。

## 二、必须收集的项目信息

写部署环节前，先确认：

```text
项目名称：______
项目类型：网页 / H5 / 工具 / API / 课程项目 / 其他
GitHub 链接：有 / 无，链接：______
README 文件：有 / 无，路径或链接：______
目标部署平台：GitHub Pages / Vercel / Netlify / Cloudflare / 飞书妙搭 / 其他
部署工具：Codex / WorkBuddy / Claude Code
最终验收：打开链接能看到什么、能操作什么、截图证明什么
```

如果项目有 GitHub 链接，教程必须把链接放进部署提示词中。

如果项目有 README，必须读取 README，并把 README 加入教程正文。加入方式至少满足一种：

- 放一个“README 速读”高亮块，概括项目用途、运行方式、构建命令、环境变量和限制。
- 放 README 链接或 GitHub README 锚点。
- 如果 README 是本地文件，作为附件或独立飞书文档入口交付。

提炼内容包括：

- 项目用途
- 本地运行方式
- 构建命令
- 环境变量
- 部署说明
- 已知限制
- 许可证或公开范围

README 不是附录摆设。它必须进入教程的“部署前准备”“README 速读”和“交付提示词”。不要只在内部读过却不放进教程。

## 三、自然语言部署提示词

部署环节优先提供可复制的自然语言提示词。模板：

```text
你是我的部署搭档。请先阅读这个项目的 README 和文件结构，不要急着改代码。

项目 GitHub 链接：______
项目目标：______
我希望部署到：______
请你先判断：
1. 这个项目是什么类型。
2. README 里写了哪些运行和构建要求。
3. 当前最适合的部署方式是什么。
4. 还缺哪些环境变量、权限或配置。

然后请按下面顺序执行：
1. 先给我部署计划。
2. 确认无误后再修改必要文件。
3. 部署完成后返回访问链接。
4. 用浏览器打开链接截图验证。

不要讲解无关代码细节。只解释部署必须知道的决策、风险和验证结果。
```

如果没有 GitHub 链接，改成：

```text
这个项目目前只有本地文件。请先读取 README 和文件结构，判断能否部署。
如果需要 GitHub 仓库，请告诉我需要创建仓库、推送哪些文件、忽略哪些文件。
```

## 四、教程正文里的部署章节结构

部署章节必须有前言，承接前文成果：

```text
上一节你已经把项目做出来了。现在的问题不是继续加功能，而是把它交给别人也能打开。部署这一步，我们不讲代码细节，只讲怎么把项目交给 AI 工具完成上线，并用截图验证结果。
```

推荐结构：

```text
H1：把项目交给 AI 工具部署
  Callout：这一章的目标和注意事项
  H2：部署前先准备三样东西
    Callout：缺 GitHub / README / 环境变量时先不要部署
    H3：GitHub 链接
    H3：README 文件
    H3：最终验收标准
  H2：复制这段自然语言部署提示词
    Callout：不要让 AI 直接改代码，先让它给部署计划
    H3：给 Codex 的版本
      H4：提示词正文
      H4：让 Codex 先返回部署计划
      H4：确认后再执行与截图验证
    H3：给 WorkBuddy 的版本
      H4：提示词正文
      H4：让 WorkBuddy 读取 README 与项目结构
      H4：确认后再执行与截图验证
    H3：给 Claude Code 的版本
      H4：提示词正文
      H4：让 Claude Code 先列风险与缺失配置
      H4：确认后再执行与截图验证
  H2：部署后怎么验收
    Callout：命令成功不等于用户能打开
    H3：打开线上链接
      H4：网页能正常访问
      H4：关键功能能操作
    H3：截图标注关键区域
      H4：红框圈出线上 URL
      H4：红色箭头指向成功状态
    H3：记录问题和下一轮反馈
      H4：把错误截图发给 AI 工具
      H4：要求 AI 给修复计划
```

项目部署教程必须使用 H1/H2/H3/H4 四级结构。H4 不是凑层级，主要用于承载同一个 H3 下的细分动作，例如“提示词正文 / 先给计划 / 执行验证 / 截图标注 / 下一轮反馈”。只有在单页极短教程中才允许退到 3 个层级，并必须说明原因。

每个标题层级下都要有高亮块提示注意事项、目标或常见坑。

每个 H1 / H2 章节开头都必须有前言段：

```text
上一节解决了 ______。这一节要把 ______ 交给 AI 工具处理。学完后你会得到 ______，下一节会用它来 ______。
```

前言要承上启下，不能只写“本节介绍部署”。

## 五、截图要求

部署环节必须有截图：

- GitHub 项目页或 README 截图。
- AI 工具生成部署计划的对话截图。
- 部署平台成功页面或线上访问页面截图。
- 最终网页可访问截图。

截图必须来自真实网页或工具界面，优先使用 computer use、浏览器控制、Playwright、Chrome 等可用插件。

如果当前会话没有直接暴露截图插件，先检索或启用可用能力，例如 `computer-use`、浏览器控制、Chrome、Playwright 或其他截图插件；能调用时必须实际调用，不能用“截图占位”替代真实网页截图。只有权限、登录、网络或工具不可用时，才允许说明原因并改用画板/表格替代。

截图标注统一使用红色画笔：

- 红框圈出 GitHub 链接、README 关键段落、部署成功状态、线上 URL。
- 红色箭头指向“下一步要看哪里”。
- 数字序号与教程中的有序步骤一致。

## 六、验收清单

部署章节完成前检查：

- [ ] 已读取项目 README，或明确说明项目没有 README。
- [ ] README 已作为“README 速读”高亮块、链接、附件或独立飞书文档入口加入教程。
- [ ] 如有 GitHub 链接，已写入教程和部署提示词。
- [ ] 部署提示词是自然语言，不是代码细节讲解。
- [ ] 分别说明 Codex / WorkBuddy / Claude Code 可如何接手。
- [ ] 已写明部署前准备、部署计划、执行、验证四个阶段。
- [ ] 已规划真实网页截图，并要求红色画笔标注。
- [ ] 已说明最终访问链接如何验收。
- [ ] 已使用 H1/H2/H3/H4 四级结构；如短教程只使用 3 个层级，已说明原因。
- [ ] H4 层级承载了真实细分动作，不是为了凑层级硬加标题。
- [ ] 每个 H1/H2 章节都有承上启下的前言说明。
- [ ] 每个层级都加入高亮注意事项、目标或常见坑。
