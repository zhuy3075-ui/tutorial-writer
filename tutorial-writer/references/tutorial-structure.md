# tutorial-structure.md — 三种教程骨架

根据教程内容类型，选择一种骨架。不要混用，不要自创。

---

## 骨架一：操作指南型

**适用场景**：教一个具体操作流程——"怎么做 X"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备 / 🖥 环境

———

步骤 1：[名称]
  这一步解决什么问题 / 为什么先做这一步
  做这个 → 你会看到这个
  [图：标注位置]

步骤 2：[名称]
  上一步完成后，为什么轮到这一步
  做这个 → 你会看到这个
  [图：标注位置]

步骤 3：[名称]
  这一步如何把前面的结果变成可验证产出
  做这个 → 你会看到这个
  [图：标注位置]

———

🎉 最终效果
  [图：完成后的界面]

❌ 常见问题
  | 问题 | 原因 | 解决 |
  |------|------|------|
  | ...  | ...  | ...  |

———

📝 练习
  打开 X → 做 Y → 你会看到 Z

🏁 能力变化回收
  学之前你只能 ______
  学完后你能 ______
  判断自己掌握的标准是 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用的 prompt、命令、模板]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交
  提交方式：______

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- 步骤不超过 5 步。超过说明该拆成两篇
- 每个步骤前必须解释“为什么现在做这一步”，不能只列动作
- 每步一图，图不能省
- "常见问题"必须来自真实踩坑，不是想象
- 实操作业**不是必加**——只在技能型章节（有操作、有产出）添加。纯概念讲解可不加
- 实操作业必须给现成材料：prompt 源码、命令模板、示例文件——不给"请自行编写"

---

## 骨架二：概念讲解型

**适用场景**：教一个抽象概念——"什么是 X""为什么需要 X"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备

———

💡 一个例子
  [具体场景，读者能代入的]

🔍 这个例子暴露了什么问题
  [从例子里抽出读者真正卡住的点]

📖 它是什么
  [定义 + 类比]
  [图：概念关系图，如适用]

🔗 和你有什么关系
  [回到读者的工作场景]

———

试试看
  [5 分钟内可验证的小实验]

🗣 停下来讨论
  你遇到过这种情况吗？

———

📝 练习
  用今天学的概念，解释你项目中遇到的一个问题

🏁 能力变化回收
  学之前你以为 ______
  学完后你能区分 ______ 和 ______
  判断自己理解的标准是 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- "一个例子"必须来自读者的日常经验，不是技术圈的例子
- 从例子到定义之间必须有“这个例子暴露了什么问题”，避免硬切概念
- "和你有什么关系"是这篇教程的魂——没这一段，概念就是空中楼阁
- 讨论节点必埋
- 实操作业不强制，加不加看"这个概念有没有一个能立刻用的落地场景"

---

## 骨架三：案例拆解型

**适用场景**：拆一个真实案例——"我是怎么做 X 的""这个项目为什么成功/失败"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备

———

📋 背景
  谁、什么场景、遇到了什么问题

💥 遇到的问题
  [具体描述，不是"系统性能不足"这种虚的]
  [图：问题截图/示意图]

🧱 为什么常规办法不够
  [解释为什么不能直接套模板 / 为什么旧方法会失败]

🔧 怎么解决的
  尝试 1：______ → 结果：______ ❌
  尝试 2：______ → 结果：______ ✅
  [图：解决方案对比]

💡 为什么这样解决
  [决策逻辑，不是"因为好用"]

———

✂️ 你能抄什么
  | 场景 | 这个思路能用吗 | 需要调整什么 |
  |------|---------------|-------------|
  | ...  | ...           | ...         |

🗣 停下来讨论
  你们项目里有没有类似的情况？

———

📝 练习
  把你项目里一个类似的问题，套这个解决思路试一下

🏁 能力变化回收
  学之前你只能照抄案例
  学完后你能判断 ______ 场景是否适合复用
  不能复用时，你知道要调整 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- "遇到的问题"要给具体数字："页面加载 8 秒""月活卡在 200 上不去"——不是"性能瓶颈"
- 从问题到解决方案之间必须解释“为什么常规办法不够”
- "怎么解决的"必须包含失败尝试。没有失败路径的案例是广告
- "你能抄什么"表格是对读者最有价值的部分——花时间写满
- 实操作业不强制，加不加看"案例里的方法能不能让学员立刻套用"

---

## 骨架选择决策树

```
教程要教什么？
  ├─ 一个具体操作 → 骨架一（操作指南型）
  ├─ 一个抽象概念 → 骨架二（概念讲解型）
  └─ 一个真实案例 → 骨架三（案例拆解型）
```
