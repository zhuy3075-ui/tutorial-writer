# tutorial-structure.md — 四种教程骨架

根据教程内容类型，选择一种骨架。不要混用，不要自创。

---

## 骨架一：操作指南型

**适用场景**：教一个具体操作流程——"怎么做 X"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备 / 🖥 环境

———

步骤 1：[名称]
  这一步解决什么问题 / 为什么先做这一步
  做这个 → 你会看到这个
  [图：标注位置]

步骤 2：[名称]
  上一步完成后，为什么轮到这一步
  做这个 → 你会看到这个
  [图：标注位置]

步骤 3：[名称]
  这一步如何把前面的结果变成可验证产出
  做这个 → 你会看到这个
  [图：标注位置]

———

🎉 最终效果
  [图：完成后的界面]

❌ 常见问题
  | 问题 | 原因 | 解决 |
  |------|------|------|
  | ...  | ...  | ...  |

———

📝 练习
  打开 X → 做 Y → 你会看到 Z

🏁 能力变化回收
  学之前你只能 ______
  学完后你能 ______
  判断自己掌握的标准是 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用的 prompt、命令、模板]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交
  提交方式：______

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- 步骤不超过 5 步。超过说明该拆成两篇
- 每个步骤前必须解释“为什么现在做这一步”，不能只列动作
- 每步一图，图不能省
- "常见问题"必须来自真实踩坑，不是想象
- 实操作业**不是必加**——只在技能型章节（有操作、有产出）添加。纯概念讲解可不加
- 实操作业必须给现成材料：prompt 源码、命令模板、示例文件——不给"请自行编写"

---

## 骨架二：概念讲解型

**适用场景**：教一个抽象概念——"什么是 X""为什么需要 X"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备

———

💡 一个例子
  [具体场景，读者能代入的]

🔍 这个例子暴露了什么问题
  [从例子里抽出读者真正卡住的点]

📖 它是什么
  [定义 + 类比]
  [图：概念关系图，如适用]

🔗 和你有什么关系
  [回到读者的工作场景]

———

试试看
  [5 分钟内可验证的小实验]

🗣 停下来讨论
  你遇到过这种情况吗？

———

📝 练习
  用今天学的概念，解释你项目中遇到的一个问题

🏁 能力变化回收
  学之前你以为 ______
  学完后你能区分 ______ 和 ______
  判断自己理解的标准是 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- "一个例子"必须来自读者的日常经验，不是技术圈的例子
- 从例子到定义之间必须有“这个例子暴露了什么问题”，避免硬切概念
- "和你有什么关系"是这篇教程的魂——没这一段，概念就是空中楼阁
- 讨论节点必埋
- 实操作业不强制，加不加看"这个概念有没有一个能立刻用的落地场景"

---

## 骨架三：案例拆解型

**适用场景**：拆一个真实案例——"我是怎么做 X 的""这个项目为什么成功/失败"。

### 结构

```
⏱ 时间 / 🎯 目标 / 📋 准备

———

📋 背景
  谁、什么场景、遇到了什么问题

💥 遇到的问题
  [具体描述，不是"系统性能不足"这种虚的]
  [图：问题截图/示意图]

🧱 为什么常规办法不够
  [解释为什么不能直接套模板 / 为什么旧方法会失败]

🔧 怎么解决的
  尝试 1：______ → 结果：______ ❌
  尝试 2：______ → 结果：______ ✅
  [图：解决方案对比]

💡 为什么这样解决
  [决策逻辑，不是"因为好用"]

———

✂️ 你能抄什么
  | 场景 | 这个思路能用吗 | 需要调整什么 |
  |------|---------------|-------------|
  | ...  | ...           | ...         |

🗣 停下来讨论
  你们项目里有没有类似的情况？

———

📝 练习
  把你项目里一个类似的问题，套这个解决思路试一下

🏁 能力变化回收
  学之前你只能照抄案例
  学完后你能判断 ______ 场景是否适合复用
  不能复用时，你知道要调整 ______

🛠 实操作业（如本章有必要）
  任务：______
  用到的工具/提示词：
  \`\`\`
  [直接可复制使用]
  \`\`\`
  ✅ 合格标准：
  - [ ] ______
  - [ ] ______
  🔍 自检：
  做完后对照合格标准逐项打勾，全勾了再提交

← 前置阅读：[标题]
下一步：[标题] →
```

### 要点

- "遇到的问题"要给具体数字："页面加载 8 秒""月活卡在 200 上不去"——不是"性能瓶颈"
- 从问题到解决方案之间必须解释“为什么常规办法不够”
- "怎么解决的"必须包含失败尝试。没有失败路径的案例是广告
- "你能抄什么"表格是对读者最有价值的部分——花时间写满
- 实操作业不强制，加不加看"案例里的方法能不能让学员立刻套用"

## 骨架四：公众号项目实战型

**适用场景**：写一篇像公众号图文长文的项目实战教程——"我用某个 AI 工具做出了 X，过程怎么拆，别人怎么复用"。

这种骨架适合工具体验、Agent 项目复盘、H5/网页/图片/视频/自动化这类完整小项目。它不是冷冰冰的步骤文档，而是有真实语气、有成果展示、有过程拆解、有方法迁移的图文教程。

### 结构

```text
👋 轻松开场
  我是谁 / 为什么今天讲这个 / 这次做了什么

🎬 先看最终效果
  [图/录屏/截图：最终成品]
  这个项目最终包含哪些能力：
  1. ______
  2. ______
  3. ______

💡 如果你也想做，先往下看
  用 1 段话说明适合谁、能复用到什么场景

———

🧭 我是怎么从想法拆到项目的
  模糊想法：______
  拆成页面 / 素材 / 内容 / 交互 / 部署
  [图：项目拆解流程图或思维导图]

📋 开工前准备清单
  | 类别 | 需要准备什么 | AI 能不能帮你做 |
  |------|--------------|------------------|

🧩 阶段 1：把想法变成产品规划
  先用自然语言告诉 AI 你要做什么
  AI 反问了哪些关键问题
  你补充了哪些偏好
  [图：规划文档/对话截图]

🧩 阶段 2：准备内容和素材
  题库 / 文案 / 图片 / 视频 / 音效 / 字体分别怎么处理
  哪些让 AI 生成，哪些自己确认
  [图：素材清单或资产方案]

🧩 阶段 3：生成项目并预览
  AI 如何把前面的规划落到页面、文件和部署里
  你看到预览后怎么反馈
  [图：预览页/部署结果]

🧩 阶段 4：根据反馈打磨
  不是一次成型，而是用具体反馈继续改
  反馈要落到页面、素材、文案、间距、动效这些具体对象

———

🕳 我这次踩到的坑
  具体问题：______
  当时为什么会发生：______
  下次应该提前怎么做：______

🔍 这个工具到底适合什么
  它不是只会聊天，而是能围绕目标推进项目
  | 能力 | 在项目里怎么体现 | 适合什么任务 |
  |------|------------------|--------------|

🌱 你可以怎么复用
  把世界杯 H5 换成：知识竞赛 / 品牌活动 / 课程测验 / 节日小游戏
  每种场景要改什么

📝 5 分钟练习
  选一个你想做的小项目，用一句话写清：
  目标用户：______
  最终效果：______
  必备页面/素材：______
  你希望 AI 先问你的 3 个问题：______

🏁 小结
  这篇不是让你记住某个工具按钮，而是学会：
  模糊想法 → 项目规划 → 素材清单 → 生成预览 → 反馈打磨 → 可分享成品
```

### 要点

- 开头可以有轻松个人语气，但不要闲聊超过 150 字。
- 必须先展示最终效果，再讲过程。读者先知道"做出来长什么样"，才愿意看步骤。
- 每个阶段都要有图：最终效果图、流程图、清单图、对话截图、预览截图至少选 3 类。
- 自然语言 prompt 比代码更重要。除非必须复制执行，否则不要贴长代码。
- 每阶段都写"我怎么说给 AI 听"，而不是只写"点击哪里"。
- 必须写 AI 反问、用户补充偏好、反馈打磨这些互动过程，体现 Agent 项目协作感。
- 工具介绍放在项目过程之后。先让读者看到工具办成了什么，再解释它是什么。
- 小结要上升到可迁移方法，不停留在"这个工具很好用"。

---

## 骨架选择决策树

```
教程要教什么？
  ├─ 一个具体操作 → 骨架一（操作指南型）
  ├─ 一个抽象概念 → 骨架二（概念讲解型）
  ├─ 一个真实案例 → 骨架三（案例拆解型）
  └─ 一个完整小项目 / 工具体验 / Agent 项目复盘 → 骨架四（公众号项目实战型）
```
