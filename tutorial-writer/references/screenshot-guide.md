# screenshot-guide.md — 截图策略

## 什么时候截图

判断标准只有一个：**读者不看这张图，操作会卡住吗？** 会，就截。不会，就不截。

截图优先来自真实网页或真实工具界面，不用想象图替代。需要操作网页、后台、部署平台、GitHub、飞书、Codex、WorkBuddy 或 Claude Code 时，必须优先调用当前可用的 `computer use`、浏览器控制、Playwright、Chrome 或其他截图插件获取真实截图。

如果当前工具列表里没有直接可用的截图能力，先检索或启用可用插件/能力，例如 `computer-use`、浏览器控制、Chrome、Playwright 或环境内置截图工具。只要能调用，就必须真实打开网页或工具界面并截图；不要只写“此处截图”作为最终产物。

## 截图、画板、img2.0 图片的分工

截图负责回答“界面上在哪里点、点完看到什么”。画板和表格负责回答“逻辑上为什么这样做、两种方案怎么选、流程怎么走”。img2.0 图片负责“让抽象知识更容易理解、让学习节奏更轻松、让阶段总结一眼看懂”。

不要让截图和画板重复表达同一件事：

| 内容 | 用截图 | 用画板 / 表格 | 用 img2.0 图片 |
|------|--------|---------------|----------------|
| 按钮在哪里 | ✅ | ❌ | ❌ |
| 操作前后界面变化 | ✅ | 可辅助 | ❌ |
| 工具协作流程 | ❌ | ✅ | 可做轻量类比 |
| 概念层级关系 | ❌ | ✅ | 可做总结卡 |
| 错误做法 vs 正确做法 | 可截界面 | ✅ | 可做情境演示 |
| 决策路径 | ❌ | ✅ | ❌ |
| 学习压力缓冲 | ❌ | ❌ | ✅ |
| 抽象知识类比 | ❌ | 可辅助 | ✅ |

如果一张图既想说明界面，又想说明抽象逻辑，拆成两张：先截图告诉读者点哪里，再用画板说明为什么这样走。如果还需要缓解学习压力或做类比，再加 img2.0 图片。

截图数字序号必须和正文步骤编号一致。正文写“步骤 2”，截图里不要标“①”从头重排导致读者对不上。

### 三种必须截图

#### 1. 需要点击的位置

读者需要点一个按钮、选一个菜单、填一个输入框——必须截。

标注方式：红框圈出目标区域 + 数字序号标注操作顺序。所有画笔、框线、箭头统一使用红色。

#### 2. 操作前后对比

做某一步之前 vs 之后，界面发生了什么变化——必须截。

标注方式：两张图并排，箭头标注变化点。箭头使用红色。

#### 3. 容易卡住的界面

你踩过的坑——"做到这一步你可能会看到这个错误/这个界面，别慌，是因为______"——必须截。

标注方式：箭头指向异常处 + 文字说明原因。箭头和圈注使用红色。

## 标注规范

只用三种标注元素，不用更多：

| 元素 | 用途 | 示例 |
|---|---|---|
| 红框 | 圈出目标区域 | 圈按钮、输入框、关键信息 |
| 数字序号 | 标注操作顺序 | ① ② ③，按点击顺序排 |
| 箭头 | 指向具体位置 + 表示变化 | 从旧值指向新值，从疑惑指向解释 |

禁止使用：花哨边框、渐变高亮、动画箭头、表情贴纸。

颜色规则：

- 画笔、红框、箭头、圈注统一使用红色。
- 不使用多色标注，避免读者不知道先看哪里。
- 如果截图背景本身偏红，仍使用红色标注，但加粗线条或加白色底衬。

步骤截图规则：

- 每个明确操作步骤尽量对应一张工具截图。
- 截图要来自该步骤真正打开的网页或工具界面。
- 需要网页证据时，必须先在浏览器、Playwright、Chrome 或 computer use 中打开对应 URL，再截图。
- 截图前写一句“这张图证明什么”，截图后写一句“下一步看哪里”。
- 无法截图时必须说明原因，并用画板、表格或文字检查项替代。

## 配图位标注格式

教程正文中不嵌入实际图片，而是标注配图位。格式：

```
[图：Photoshop 菜单栏 → 滤镜 → 锐化 → USM 锐化，红框圈出 USM 锐化选项]
```

执行生成时，将这些标注替换为截图。

---

## 不用动图

GIF 的问题：
- 速度不可控，读者看不清
- 无法暂停看某一帧
- 加载慢

替代方案：2-3 张静图并排 + 箭头标注流程走向。

---

## 截图环境要求

- 截图时桌面整洁（无关窗口最小化）
- 界面使用默认主题（不用高对比度或暗色模式，除非教程主题需要）
- 截图分辨率不低于 2x 视网膜
- 网页截图优先在浏览器中截取当前页面；不要用生成图模拟网页界面。
- 截图文件保存到教程资产目录，并在提示词或交付说明中记录来源页面、截图时间和对应步骤。
